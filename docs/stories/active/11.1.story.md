## Story
**Como** usuario dono de cegonha no publico 55+ do FrotaViva,
**Quero** que todas as telas do dashboard tenham cores coerentes em light e dark mode, sem badges quebrados ou cards com contraste errado,
**Para que** eu confie visualmente no produto e nao desconfie de que "tem alguma coisa errada nessa tela".

## Acceptance Criteria

1. O arquivo `app/globals.css` contem a escala completa de `--c-primary-50` ate `--c-primary-900` tanto para light mode quanto para dark mode. Nenhum shade de primary usado em classe Tailwind fica sem definicao.
2. O arquivo `app/globals.css` contem tokens de badge: `--c-badge-warning-bg`, `--c-badge-warning-fg`, `--c-badge-info-bg`, `--c-badge-info-fg`, `--c-badge-neutral-bg`, `--c-badge-neutral-fg`, `--c-badge-success-bg`, `--c-badge-success-fg`, `--c-badge-danger-bg`, `--c-badge-danger-fg`, cada um com valor para light e dark mode.
3. As 231 classes Tailwind nativas (`amber-*`, `slate-*`, `red-*`, `gray-*` com numero) em componentes do dashboard sao substituidas por tokens do design system (primary, neutral, badge-*, alert-*). A busca global por essas classes em `components/` e `app/(dashboard)/` retorna zero resultados (a nao ser em `globals.css` onde os tokens sao definidos).
4. O modo dark funciona coerente em: BI (todas as 4 paginas), fechamentos (lista e detalhe), viagens (lista e form), gastos, dashboard home, pagina de empresa, pagina de motoristas, pagina de caminhoes.
5. Contraste minimo 4.5:1 para texto normal e 3:1 para texto grande em light e dark mode. Nao precisa auditar tela por tela com tool, mas os badges de status devem ter contraste visivel em ambos os modos.
6. `npm run typecheck && npm run lint && npm run build` passam.
7. Nenhuma classe Tailwind zumbi nova e introduzida (ex: `bg-c-alert-warning-bg0` que nao existe). Se o @dev usar uma classe custom, ela deve estar definida no globals.css.
8. Um arquivo `docs/design-system/tokens.md` (ou atualizacao de doc existente) lista todos os tokens disponiveis e quando usar cada um.

## Dev Notes

- Atencao a regra do Tailwind v4 (memoria do usuario): usar `--c-*` para tokens custom, nao `--color-*` que gera wildcards. Ja existe essa convencao no projeto, manter.
- Arquivos previsivelmente afetados: `app/globals.css`, `components/ui/**`, `components/viagens/**`, `components/fechamentos/**`, `components/bi/**`, `components/dashboard/**`, `components/empresa/**`, `components/motorista/**`, `components/caminhao/**`, `app/(dashboard)/**/page.tsx`.
- NAO refatorar componentes alem da substituicao de cores. Qualquer mudanca estrutural vai em story separada (Epic D).
- A story 11.2 (lint rule) vai bloquear a reintroducao. Sem ela, o trabalho desta story decai rapido. Ambas devem ser feitas no mesmo sprint.
- Cuidado com componentes que usam cor para transmitir informacao (ex: badge verde = pago). Apos substituir token, conferir visualmente se ainda da pra distinguir estados.
