## Story
**As a** motorista que acabou de receber credenciais temporarias do patrao,
**I want** ser informado de forma clara que minha senha foi criada pelo proprietario e que e recomendado troca-la,
**so that** eu possa tomar uma acao consciente de seguranca sem ser bloqueado de usar o sistema.

## Acceptance Criteria

1. Existe um mecanismo para detectar se o usuario esta fazendo seu primeiro login com senha temporaria: o campo `force_password_change` na tabela `usuario` (coluna boolean, default `false`) e definido como `true` durante o cadastro atomico da Story 8.1.
2. A migration adiciona a coluna `force_password_change BOOLEAN NOT NULL DEFAULT false` na tabela `usuario`.
3. A server action `createMotoristaComConta` (Story 8.1) define `force_password_change = true` ao inserir o registro em `usuario`.
4. Apos login bem-sucedido, se o usuario autenticado tiver `usuario.force_password_change = true`, e exibido um banner persistente no topo da pagina (dentro do layout do dashboard) com a mensagem: "Sua senha foi criada pelo proprietario. Recomendamos que voce a troque para maior seguranca."
5. O banner contem o botao "Trocar agora" que redireciona para `/perfil`.
6. O banner contem o botao "Depois" que dispensa o banner para a sessao atual (nao permanente).
7. Dispensar o banner com "Depois" NAO altera `force_password_change` no banco -- o banner voltara na proxima sessao.
8. Ao trocar a senha em `/perfil` com sucesso, a server action `alterarSenha` define `force_password_change = false` na tabela `usuario` para o usuario autenticado.
9. Apos `force_password_change = false`, o banner nao e mais exibido em nenhuma sessao futura.
10. O banner e visivel apenas para usuarios com `force_password_change = true` -- usuarios com senha propria nao veem o banner.
11. O banner NAO bloqueia nenhuma funcionalidade -- o usuario pode navegar e usar o sistema normalmente.
12. O banner tem estilo de aviso (amarelo/laranja) para indicar acao recomendada, sem ser alarme (nao vermelho/critico).

## Dev Notes

### Deteccao de Primeiro Login
A abordagem e via campo no banco `force_password_change` na tabela `usuario`, verificado no layout do dashboard a cada carga de pagina (Server Component). Isso garante consistencia entre sessoes e dispositivos.

Alternativa (NAO usar): flag em localStorage ou cookie. Seria descartado ao limpar o browser e nao funcionaria em multiplos dispositivos.

### Migration SQL
Arquivo novo: `supabase/migrations/{timestamp}_add_force_password_change.sql`
```sql
ALTER TABLE usuario ADD COLUMN IF NOT EXISTS force_password_change BOOLEAN NOT NULL DEFAULT false;
```
Usar timestamp no formato `YYYYMMDDHHMMSS` para ordenacao correta.

### Carregar no Layout (Server Component)
O `app/(dashboard)/layout.tsx` ja e um Server Component que busca dados do usuario. Adicionar a consulta do campo:
```typescript
const { data: usuario } = await supabase
  .from('usuario')
  .select('force_password_change')
  .eq('auth_id', user!.id)
  .single();
```
E passar como prop para o componente `PrimeiroLoginBanner`.

### Componente Banner (Client Component)
O banner precisa de estado local para o botao "Depois":
```typescript
'use client';
export function PrimeiroLoginBanner({ mostrar }: { mostrar: boolean }) {
  const [dispensado, setDispensado] = useState(false);
  if (!mostrar || dispensado) return null;
  // render banner
}
```

### Integracao com Story 8.1 e 8.4
Esta story depende de:
- **Story 8.1**: `createMotoristaComConta` deve definir `force_password_change = true` (AC 3)
- **Story 8.4**: `alterarSenha` deve definir `force_password_change = false` (AC 8)

Se as stories anteriores ja estiverem implementadas sem este campo, adicionar as atualizacoes como parte desta story.

### Posicionamento do Banner
O banner deve aparecer no topo do conteudo principal, abaixo do header/navbar, mas acima do conteudo da pagina. No layout atual, identificar onde o `{children}` e renderizado e inserir o banner imediatamente antes.
