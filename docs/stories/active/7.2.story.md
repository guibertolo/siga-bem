## Story
**As a** usuario do FrotaViva com multiplas empresas,
**I want** ver uma tela de selecao de empresa logo apos fazer login,
**so that** eu possa escolher em qual CNPJ/empresa quero operar naquela sessao antes de acessar o dashboard.

## Acceptance Criteria

1. Apos autenticacao, o sistema consulta `fn_get_user_empresas()`: se o usuario tem mais de 1 empresa ativa vinculada, redireciona para `/selecionar-empresa`; se tem exatamente 1, vai direto ao `/dashboard` (bypass automatico).
2. A tela `/selecionar-empresa` lista todas as empresas ativas vinculadas ao usuario em formato de cards grandes (acessivel para publico 60+).
3. Cada card exibe: razao_social, CNPJ formatado (XX.XXX.XXX/XXXX-XX), nome_fantasia (se houver), e o role do usuario naquela empresa (ex: "Dono", "Administrador", "Motorista").
4. Empresas inativas aparecem com badge visual "Inativa" e botao "Entrar" desabilitado (nao clicavel).
5. Ao clicar "Entrar" em um card, o sistema chama a server action `switchEmpresa(empresaId)` que internamente usa `fn_switch_empresa()`, faz `revalidatePath('/')` e redireciona para `/dashboard`.
6. A ultima empresa usada aparece com badge "Ultimo acesso" e fica primeiro na lista (ordem: ultima usada primeiro, demais por razao_social ascendente).
7. Usuario com 1 empresa nunca ve esta tela — o fluxo e identico ao atual (AC: 1).
8. A tela exibe botao "+ Nova Empresa" que leva para o fluxo de cadastro (Story 7.4). O botao esta visivel mas pode linkar para pagina em construcao ate Story 7.4 ser implementada.
9. Design responsivo, cards com padding generoso, fonte minima 16px, contraste WCAG AA. Botao "Entrar" com area de toque minima 44x44px.
10. Server action `switchEmpresa` valida que o usuario tem vinculo ativo com a empresa antes de executar o UPDATE — rejeita com erro se o vinculo nao existir.
11. Enquanto a troca de empresa esta sendo processada, o botao "Entrar" exibe estado de loading (disabled + spinner) para evitar duplo clique.

## Dev Notes

### Onde fica o redirect pos-login
Checar o arquivo de callback/redirect atual em `app/(auth)/`. A logica de "1 empresa = vai para dashboard, N empresas = vai para selecao" deve ficar no callback do Supabase Auth ou no middleware.

### fn_get_user_empresas via RPC
```typescript
const { data, error } = await supabase.rpc('fn_get_user_empresas')
```
Retorna array de `{ empresa_id, razao_social, nome_fantasia, cnpj, role, is_active }`.

### Ordenacao da ultima empresa
`ultima_empresa_id` em `usuario` guarda a ultima empresa usada. Ao listar, ordenar colocando a que tem `is_active = true` (ou `empresa_id = usuario.ultima_empresa_id`) primeiro.

### CNPJ formatting
Formatar CNPJ de `14 digitos` para `XX.XXX.XXX/XXXX-XX` no frontend.

### Role display
Traduzir: `dono` -> "Dono", `admin` -> "Administrador", `motorista` -> "Motorista".

### Dependencia
Esta story depende de 7.1 estar concluida (tabela `usuario_empresa`, funcoes SQL e types).

### Testing
- Testar fluxo com usuario de 1 empresa: deve ir direto ao dashboard
- Testar fluxo com usuario de 2 empresas: deve aparecer tela de selecao
- Testar clique em empresa inativa: botao nao deve responder
- Testar server action com empresa sem vinculo: deve retornar erro
- Testar que apos selecao os dados do dashboard sao da empresa selecionada
