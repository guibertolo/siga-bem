## Story
**As a** dono de frota com multiplas empresas,
**I want** que as consultas multi-empresa respeitem RLS nativamente,
**so that** nao haja risco de vazamento de dados entre empresas por erro de chamada sem filtro `empresa_id`.

## Epic
Epic 22 — Infra Hardening.

## Origem
Addendum 2026-04-14, item #1 (Audit Addendum — architecture-audit.md:143)

## Acceptance Criteria

1. A funcao `queryMultiEmpresa` em `lib/queries/multi-empresa-query.ts` deixa de usar `createAdminClient()` (service_role).
2. A funcao passa a usar o client autenticado do usuario (session-based).
3. Uma funcao SQL `fn_get_query_empresas(user_id uuid, selected_empresas uuid[])` e criada via migration em `supabase/migrations/`, retornando a intersecao de `selected_empresas` com os registros ativos em `usuario_empresa` para o usuario autenticado.
4. `queryMultiEmpresa` invoca `fn_get_query_empresas` e usa o resultado como lista de IDs permitidos, eliminando o filtro manual `.eq('empresa_id', eid)` espalhado nos callers.
5. Os 20 call sites em 4 arquivos (bi, dashboard, fechamentos, viagens) continuam funcionando sem alteracao de interface (refatoracao interna transparente):
   - `app/(dashboard)/bi/page.tsx` (linhas 200-210)
   - `app/(dashboard)/dashboard/page.tsx` (linhas 102-104)
   - `app/(dashboard)/fechamentos/page.tsx` (linha 59)
   - `app/(dashboard)/viagens/page.tsx` (linha 30)
6. RLS das tabelas consultadas permanece ativo — a funcao SQL opera dentro do contexto do usuario autenticado (SECURITY INVOKER ou equivalente seguro).
7. Nenhum caller pode acessar dados de empresa a qual o usuario nao pertence ou nao esta ativo (`usuario_empresa.ativo = true`).
8. Auditoria RLS ANTES da migracao: verificar que as policies RLS das tabelas consultadas por `queryMultiEmpresa` (viagem, gasto, fechamento, motorista, caminhao, abastecimento) estao ativas e corretas — documentar o estado encontrado antes de qualquer alteracao.
9. Testes de integracao por modulo: ao menos um teste por modulo (BI, dashboard, fechamentos, viagens) verificando que usuario sem acesso a empresa X nao recebe dados dela apos a refatoracao.
10. Smoke test obrigatorio antes de merge: abrir cada uma das 4 paginas (BI, Dashboard, Fechamentos, Viagens) com usuario multi-empresa e verificar que dados aparecem corretamente. Documentar no Agent Record.
11. `npm run lint && typecheck && build` passam.

## RLS Audit (AC 8) — Estado pre-migracao

Todas as tabelas consultadas por `queryMultiEmpresa` tinham RLS ativo com policies SELECT usando `empresa_id = fn_get_empresa_id()` (empresa ativa unica). Isso impedia multi-empresa com client autenticado.

| Tabela | RLS Ativo | Policy SELECT | Guarda empresa_id | Observacao |
|--------|-----------|---------------|--------------------|----|
| viagem | Sim | `viagem_select`: empresa_id = fn_get_empresa_id() + role check | Sim | OK |
| gasto | Sim | `gasto_select_dono_admin` + `gasto_select_motorista`: fn_get_empresa_id() | Sim | Policies separadas por role (Story 21.7) |
| fechamento | Sim | `fechamento_select`: fn_get_empresa_id() + role check | Sim | OK |
| motorista | Sim | `motorista_select_empresa` + `motorista_select_self` | Sim | OK |
| caminhao | Sim | `Dono e admin gerenciam caminhoes` (FOR ALL) + `Motorista visualiza` (SELECT) | Sim | FOR ALL desmembrado na migracao |
| abastecimento | N/A | Parte da tabela `gasto` (campos opcionais) | Via gasto | Herda RLS de gasto |
| fechamento_item | Sim | JOIN com fechamento.empresa_id via fn_get_empresa_id() | Via join | OK |
| viagem_veiculo | Sim | `empresa_viagem_veiculo_all` (FOR ALL) | Sim | FOR ALL desmembrado na migracao |

**Decisao:** Atualizar SELECT policies para usar `fn_user_empresa_ids()` (array de todas empresas ativas vinculadas), mantendo INSERT/UPDATE/DELETE com `fn_get_empresa_id()` (empresa ativa unica).

## Agent Record

### Decisoes de implementacao
- [AUTO-DECISION] Criar fn_user_empresa_ids() vs modificar fn_get_empresa_id() -> Funcao nova separada (reason: fn_get_empresa_id() e usada em INSERT/UPDATE/DELETE policies que DEVEM operar na empresa ativa; alterar quebraria writes)
- [AUTO-DECISION] Atualizar SELECT RLS policies vs criar views -> Atualizar policies diretamente (reason: views adicionariam camada desnecessaria; policies sao o mecanismo correto)
- [AUTO-DECISION] Desmembrar FOR ALL policies de caminhao e viagem_veiculo -> Sim (reason: FOR ALL com fn_get_empresa_id() bloqueia SELECT multi-empresa; separar SELECT permite fn_user_empresa_ids enquanto CUD mantém fn_get_empresa_id)
- [AUTO-DECISION] SECURITY INVOKER vs DEFINER para fn_get_query_empresas -> SECURITY INVOKER (reason: Story pede INVOKER; fn_user_empresa_ids ja e DEFINER e faz o trabalho pesado no RLS)
