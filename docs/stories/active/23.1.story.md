## Story
**As a** administrador tecnico do FrotaViva,
**I want** uma tabela `foto_chamada` com RLS correta, bucket `chamadas` no Storage e indices de consulta no banco,
**so that** o sistema tenha a fundacao de dados necessaria para upload e relatorio da chamada sem risco de vazamento cross-tenant.

## Epic
Epic 23 — REL (Relatorios e Upload da Chamada).

## Acceptance Criteria

1. Migration cria a tabela `foto_chamada` espelhando `foto_comprovante` (`supabase/migrations/20260328180600_create_foto_comprovante.sql`):
   - `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
   - `empresa_id UUID NOT NULL REFERENCES empresa(id) ON DELETE RESTRICT`
   - `viagem_id UUID NOT NULL REFERENCES viagem(id) ON DELETE CASCADE`
   - `storage_path TEXT NOT NULL`
   - `thumbnail_path TEXT` (nullable, para uso futuro)
   - `content_type VARCHAR(50)`
   - `size_bytes INTEGER`
   - `uploaded_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   - `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
   - `CONSTRAINT ck_chamada_size CHECK (size_bytes IS NULL OR size_bytes > 0)`

2. RLS habilitado em `foto_chamada` com 4 policies:
   - Dono/admin: `FOR ALL USING (empresa_id = fn_get_empresa_id() AND fn_get_user_role() IN ('dono', 'admin'))`
   - Motorista SELECT: apenas viagens proprias via subquery `viagem_id IN (SELECT id FROM viagem WHERE motorista_id = fn_get_motorista_id())`
   - Motorista INSERT: `WITH CHECK` que exige `empresa_id = fn_get_empresa_id()` E `viagem_id IN (SELECT id FROM viagem WHERE motorista_id = fn_get_motorista_id() AND status IN ('planejada', 'em_andamento'))`
   - Motorista UPDATE/DELETE: NENHUMA policy — bloqueio total por omissao (imutabilidade)

3. Trigger `tg_block_chamada_insert_status` como segunda camada de seguranca, bloqueando INSERT em `foto_chamada` quando `viagem.status NOT IN ('planejada', 'em_andamento')` para o role motorista:
   - Trigger `BEFORE INSERT ON foto_chamada FOR EACH ROW`
   - Checar `fn_get_user_role()`: se `'motorista'`, buscar `viagem.status` e RAISE EXCEPTION se `status NOT IN ('planejada', 'em_andamento')`
   - Dono/admin passam sem verificacao de status

4. Migration adiciona 3 colunas OCR-ready em `viagem` (todas nullable, sem valor default):
   - `chamada_carros_count INTEGER` — quantidade de carros listados na chamada
   - `chamada_embarcador TEXT` — nome da empresa embarcadora (montadora/patio)
   - `chamada_ocr_extraido_em TIMESTAMPTZ` — quando o OCR foi executado (para uso futuro)

5. COMMENT ON COLUMN documenta `viagem.valor_total`:
   - Texto: `'Valor total do frete conforme a chamada (documento fisico recebido na embarcadora). Preenchido pelo motorista no lancamento da viagem.'`

6. Bucket `chamadas` no Storage com 3 policies (INSERT/SELECT/DELETE, SEM UPDATE — imutabilidade), espelhando o padrao de `comprovantes`:
   - INSERT: `bucket_id = 'chamadas' AND (storage.foldername(name))[1] = fn_get_empresa_id()::text`
   - SELECT: `bucket_id = 'chamadas' AND (storage.foldername(name))[1] = fn_get_empresa_id()::text`
   - DELETE: `bucket_id = 'chamadas' AND (storage.foldername(name))[1] = fn_get_empresa_id()::text`
   - UPDATE: NENHUMA policy (omissao = bloqueio)

7. Indices criados para os acessos previstos nesta epic:
   - `CREATE INDEX IF NOT EXISTS idx_foto_chamada_viagem ON foto_chamada (viagem_id)`
   - `CREATE INDEX IF NOT EXISTS idx_foto_chamada_empresa ON foto_chamada (empresa_id)`
   - `CREATE INDEX IF NOT EXISTS idx_viagem_caminhao_saida ON viagem (caminhao_id, data_saida DESC)`

8. `supabase db push` aplica a migration sem erros em ambiente local.

9. Testes RLS cobrem:
   - Motorista INSERT com `viagem.status = 'planejada'` — permitido (AC: 2, 3)
   - Motorista INSERT com `viagem.status = 'em_andamento'` — permitido (AC: 2, 3)
   - Motorista INSERT com `viagem.status = 'concluida'` — bloqueado pelo trigger (AC: 3)
   - Motorista INSERT com `viagem.status = 'cancelada'` — bloqueado pelo trigger (AC: 3)
   - Motorista SELECT vendo apenas chamadas de suas proprias viagens (AC: 2)
   - Motorista UPDATE e DELETE — bloqueados por ausencia de policy (AC: 2)
   - Admin SELECT/INSERT/UPDATE/DELETE — permitidos (AC: 2)
   - Cross-tenant: admin empresa A nao ve chamadas de empresa B (AC: 2)

10. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #119. Epic L, P1, L/M.

**Tabela espelho:** `foto_chamada` segue o mesmo schema de `foto_comprovante` (`supabase/migrations/20260328180600_create_foto_comprovante.sql`), substituindo `gasto_id` por `viagem_id`. A tabela de referencia tem colunas `uploaded_at` e `created_at` — ambas obrigatorias (AC: 1). Ler a migration completa antes de criar a nova.

**Diferenca critica vs `foto_comprovante`:** motorista nao tem DELETE em `foto_chamada` (imutabilidade da chamada). Em `foto_comprovante` o motorista pode deletar. Nao copiar as policies cegamente.

**Trigger de status — logica correta:** o trigger deve bloquear quando `status NOT IN ('planejada', 'em_andamento')`. O motorista tira a foto no momento do lancamento (status = `planejada`, antes de iniciar a viagem). O enum `viagem_status` tem 4 valores: `'planejada'`, `'em_andamento'`, `'concluida'`, `'cancelada'` (confirmado em `supabase/migrations/20260328180700_create_viagem.sql`).

**Trigger de status — dupla camada:** a RLS do INSERT ja verifica status via subquery (AC: 2). O trigger (AC: 3) e a segunda camada — garante a regra mesmo que a RLS seja alterada no futuro. Implementar ambos.

**fn_get_empresa_id() retorna UUID:** confirmado em `supabase/migrations/20260328180000_create_empresa.sql`. O cast `::text` nas policies de storage (AC: 6) e necessario porque `storage.foldername(name)` retorna TEXT.

**Nomenclatura — roles:** usar `'admin'` (nao `'gestor'`) nos ACs tecnicos e migration SQL. O termo `'admin'` e o valor real na coluna `usuario.role`. `'gestor'` e apenas o termo da interface do usuario.

**Indice de data_chegada_real removido:** o indice `idx_viagem_empresa_chegada_concluida` (parcial `WHERE status='concluida'`) foi removido desta story por nao ter consumer identificado nas stories atuais. Ficara como future work quando o relatorio de viagens concluidas for implementado.

**Colunas OCR:** `chamada_carros_count`, `chamada_embarcador`, `chamada_ocr_extraido_em` nao serao populadas nesta story. Sao preparacao para OCR futuro (item #125 do roadmap). Apenas adicionar as colunas nullable.

**COMMENT ON viagem.valor_total:** usar `COMMENT ON COLUMN viagem.valor_total IS '...'` diretamente. Nao usar `ALTER TABLE ... ALTER COLUMN`. Verificar se a coluna ja existe (existe desde `20260328180700_create_viagem.sql`) — o COMMENT sobrescreve o anterior.

**Storage bucket:** incluir o `INSERT INTO storage.buckets` na migration para eliminar drift git vs dashboard (licao aprendida com bucket `comprovantes` que foi criado fora da migration — ver nota em `20260328180600_create_foto_comprovante.sql`).

**Nomenclatura do arquivo de migration:** seguir padrao `YYYYMMDDHHMMSS_create_foto_chamada_and_storage.sql`.

**Paths de referencia:**
- Schema espelho: `supabase/migrations/20260328180600_create_foto_comprovante.sql`
- Enum viagem_status: `supabase/migrations/20260328180700_create_viagem.sql`
- Funcoes auxiliares: `fn_get_empresa_id()` (retorna UUID), `fn_get_user_role()` (retorna TEXT), `fn_get_motorista_id()` (definidas em migrations iniciais, ja existem)
- Testes RLS existentes: `__tests__/` (verificar suite existente para seguir padrao)
