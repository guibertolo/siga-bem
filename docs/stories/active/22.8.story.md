# Story 22.8: CORS explicito nos route handlers

## Story
**As a** responsavel pela seguranca do FrotaViva,
**I want** que os 3 route handlers da aplicacao declarem explicitamente politica CORS restritiva,
**so that** nenhuma origem desconhecida consiga chamar esses endpoints cross-origin, eliminando o risco de adicionar `Access-Control-Allow-Origin: *` acidentalmente no futuro.

## Epic
Epic 22 — Infra Hardening.

## Origem
Addendum 2026-04-14, item #8

## Contexto Tecnico

Estado atual verificado via leitura dos arquivos:

- `app/api/auth/signout/route.ts`: exporta apenas `POST`. Nenhum header CORS. Usa `NEXT_PUBLIC_SITE_URL` para redirect.
- `app/auth/callback/route.ts`: exporta apenas `GET`. Nenhum header CORS. Fluxo interno OAuth (Supabase callback).
- `app/(dashboard)/financeiro/historico/export/route.ts`: exporta apenas `GET`. Nenhum header CORS. Retorna CSV.
- `middleware.ts`: apenas session refresh via `updateSession`. Matcher exclui `auth/`. Nenhuma logica CORS.
- `next.config.ts`: tem `async headers()` com 5 headers de seguranca (X-Frame-Options, X-Content-Type-Options, Referrer-Policy, X-DNS-Prefetch-Control, Permissions-Policy) aplicados em `source: '/(.*)'`. Nenhum header CORS.
- Grep confirmado: zero ocorrencias de `CORS`, `cors`, `Access-Control` ou `allowedOrigins` em todo o projeto.

## Acceptance Criteria

1. Um modulo utilitario `lib/cors.ts` e criado contendo:
   - A lista de origens permitidas derivada de `NEXT_PUBLIC_SITE_URL` (produção) e `http://localhost:3000` (desenvolvimento). Nenhuma outra origem e permitida.
   - Uma funcao `applyCorsHeaders(response: NextResponse, origin: string | null): NextResponse` que adiciona os headers `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods` e `Access-Control-Allow-Headers` apenas se a origem estiver na allowlist. Se a origem nao estiver na allowlist, nenhum header CORS e adicionado (o browser bloqueia por ausencia, nao por 403 explicito — comportamento correto para CORS).
   - Uma funcao `handleOptions(request: Request): NextResponse | null` que retorna resposta 204 com headers CORS corretos se o metodo for `OPTIONS` e a origem estiver na allowlist; retorna `null` em qualquer outro caso (metodo nao-OPTIONS ou origem invalida). Quando o handler recebe `null`, deve prosseguir com o fluxo normal sem adicionar headers CORS.

2. O handler `app/api/auth/signout/route.ts` e atualizado para:
   - Exportar `export async function OPTIONS(request: Request)` que delega para `handleOptions` e retorna 204 com headers CORS se origem valida, ou 405 se origem invalida.
   - Adicionar `applyCorsHeaders` na resposta do `POST` existente.
   - Metodo `POST` continua funcionando identico ao atual (signOut + redirect 302).

3. O handler `app/auth/callback/route.ts` e atualizado para:
   - Exportar `export async function OPTIONS(request: Request)` idem ao AC 2.
   - Adicionar `applyCorsHeaders` na resposta do `GET` existente.
   - Logica do `GET` continua identica (exchangeCodeForSession + redirects).

4. O handler `app/(dashboard)/financeiro/historico/export/route.ts` e atualizado para:
   - Exportar `export async function OPTIONS(request: Request)` idem ao AC 2.
   - Adicionar `applyCorsHeaders` na resposta do `GET` existente (incluindo ambos os `return` com CSV e o `return` de erro 401/500).
   - Logica do `GET` continua identica (autenticacao, filtros, CSV).

5. `NEXT_PUBLIC_SITE_URL` ja presente em `.env.example` e usada como origem permitida em producao. Nenhuma nova variavel de ambiente e introduzida.

6. Uma requisicao simulada com header `Origin: https://atacante.com` para qualquer dos 3 endpoints NAO recebe header `Access-Control-Allow-Origin` na resposta. Uma requisicao com `Origin` igual ao valor de `NEXT_PUBLIC_SITE_URL` recebe `Access-Control-Allow-Origin` com o valor exato da origem.

7. `npm run lint && npm run typecheck && npm run build` passam sem erros apos as alteracoes.

## Technical Notes

- Por que modulo centralizado e nao `next.config.ts` `headers()`: a funcao `headers()` no next.config aplica headers estaticos para todas as respostas de uma source, sem acesso ao valor do header `Origin` da requisicao. Para CORS correto, o header `Access-Control-Allow-Origin` precisa refletir a origem especifica da request (nao `*`), o que requer logica dinamica dentro do handler.

- Por que nao no `middleware.ts`: o middleware atual exclui `auth/` do matcher (linha 27). Incluir logica CORS no middleware exigiria ajustar o matcher e adicionaria acoplamento indesejado ao fluxo de session. Modulo utilitario por handler e mais explicito e testavel.

- Metodos permitidos por endpoint:
  - `signout`: `POST, OPTIONS`
  - `callback`: `GET, OPTIONS`
  - `export`: `GET, OPTIONS`

- Header `Access-Control-Allow-Headers` minimo recomendado: `Content-Type, Authorization`.

- Preflight OPTIONS deve retornar `204 No Content` (body vazio), nao 200.

- Complexidade: S / Risco S (sem alteracao de logica de negocio, sem nova dependencia externa).

## Testing Requirements

- Code review: `lib/cors.ts` exporta `applyCorsHeaders` e `handleOptions` com tipagem correta (sem `any`).
- Code review: os 3 handlers exportam `OPTIONS` e chamam `applyCorsHeaders` em todas as respostas.
- Teste manual: `curl -H "Origin: https://atacante.com" -X OPTIONS http://localhost:3000/api/auth/signout` — resposta NAO deve conter `Access-Control-Allow-Origin`.
- Teste manual: `curl -H "Origin: http://localhost:3000" -X OPTIONS http://localhost:3000/api/auth/signout` — resposta DEVE conter `Access-Control-Allow-Origin: http://localhost:3000` e status 204.
- Teste manual: `curl -H "Origin: http://localhost:3000" -X GET http://localhost:3000/financeiro/historico/export` — resposta DEVE conter `Access-Control-Allow-Origin: http://localhost:3000`.
