## Story
**As a** responsavel pela seguranca do FrotaViva,
**I want** Content Security Policy e HSTS configurados explicitamente nos headers HTTP,
**so that** o app esteja protegido contra XSS e seja elegivel para HSTS preload no navegador.

## Epic
Epic 22 — Infra Hardening.

## Origem
Addendum 2026-04-14, item #2

## Acceptance Criteria

1. O arquivo `next.config.ts` define um header `Content-Security-Policy-Report-Only` com politica CSP completa cobrindo: `default-src`, `script-src`, `style-src`, `img-src`, `connect-src`, `font-src`, `object-src 'none'`, `frame-ancestors 'none'`.
2. O header CSP opera em modo `report-only` durante pelo menos 1 semana apos o deploy (sem bloquear requisicoes reais).
3. Um endpoint de report configurado para capturar violacoes durante o periodo report-only: usar `report-to` com o header `Reporting-Endpoints` (nao `report-uri` deprecated). Endpoint externo preferido: Sentry ou `/api/csp-report` interno.
4. O header `Strict-Transport-Security` e adicionado com valor `max-age=63072000; includeSubDomains; preload`.
5. Os headers ja existentes (`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `X-DNS-Prefetch-Control`) sao preservados sem alteracao.
6. Apos 1 semana de monitoramento sem violacoes criticas, o Dev (em nova story ou PR separado) troca `Content-Security-Policy-Report-Only` para `Content-Security-Policy` — este passo esta FORA do escopo desta story (documentar como proximo passo).
7. `npm run lint && typecheck && build` passam.
8. Teste: response headers da rota `/` incluem `Strict-Transport-Security` e `Content-Security-Policy-Report-Only`.

## Technical Notes

- Arquivo a modificar: `next.config.ts`, secao `headers()`.
- Headers atuais presentes no codigo (verificado em `next.config.ts`): `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `X-DNS-Prefetch-Control`, `Permissions-Policy`. Nao remover nenhum.
- Dev Note — enforce mode: apos 1 semana de monitoramento sem violacoes criticas, criar story separada 22.2b (backlog) para trocar `Content-Security-Policy-Report-Only` para `Content-Security-Policy` em modo enforce. Nao implementar nesta story.
- Supabase, Vercel Analytics, Google Fonts e quaisquer CDNs usados pelo app devem estar no allow-list da CSP. Verificar imports no codebase antes de escrever a politica.
- Referencia HSTS preload: https://hstspreload.org — `max-age` minimo e 31536000 (1 ano), esta story usa 63072000 (2 anos) conforme instrucao do produto.
- Modo report-only nao afeta usuarios — e seguro fazer deploy imediato.
- Complexidade: M / Risco M (politica CSP errada pode quebrar recursos em enforce mode — por isso report-only primeiro).

## Testing Requirements

- Smoke test: `curl -I https://<dominio>/` e verificar presenca dos headers `Strict-Transport-Security` e `Content-Security-Policy-Report-Only`.
- Verificar no browser DevTools (Network > response headers) que CSP nao bloqueia nenhum recurso em report-only.
- Verificar que endpoint de report recebe requests (ou Sentry captura violacoes) apos deploy.
