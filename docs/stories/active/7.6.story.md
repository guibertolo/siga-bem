## Story
**As a** dono ou administrador de uma empresa no FrotaViva,
**I want** convidar um usuario existente para operar na minha empresa,
**so that** eu possa colaborar com pessoas que ja tem conta no sistema sem precisar criar contas duplicadas.

## Acceptance Criteria

1. Na tela de usuarios da empresa (`/usuarios`), o dono pode convidar um usuario existente pelo email. O campo de convite aceita um email e um role destino (admin ou motorista).
2. O dono pode convidar com role `admin` ou `motorista`. O admin pode convidar apenas com role `motorista`.
3. Ao submeter o convite: o sistema busca o usuario pelo email em `usuario` (JOIN em `auth.users` pelo email).
4. Se o usuario existir no sistema: criar registro em `usuario_empresa (usuario_id, empresa_id, role)` com `ativo = true`. Exibir confirmacao "Usuario adicionado com sucesso".
5. Se o usuario NAO existir no sistema: exibir mensagem orientando o usuario a se cadastrar primeiro. Nao enviar email automaticamente nesta story (escopo futuro).
6. Se o usuario ja tiver vinculo ativo com a empresa: exibir mensagem "Este usuario ja tem acesso a esta empresa" e nao duplicar o vinculo.
7. O vinculo criado tem o role definido pelo convidante (AC: 2) — o role nao pode ser elevado pelo proprio convidado.
8. Somente dono e admin da empresa ativa podem acessar a funcionalidade de convite. Motoristas nao veem o botao de convite.
9. A operacao de INSERT em `usuario_empresa` e protegida pela RLS policy `ue_insert_dono` criada em 7.1, garantindo que apenas dono da empresa ativa pode executar.
10. Apos adicionar o vinculo, o usuario convidado passa a ver a nova empresa na tela de selecao de empresa no proximo login.

## Dev Notes

### Buscar usuario por email
O email esta em `auth.users` (Supabase Auth), nao diretamente em `usuario`. Usar:
```typescript
// Via Supabase Admin (server-side)
const { data } = await supabaseAdmin.auth.admin.listUsers()
// ou buscar via view/funcao que expoe email seguramente
```
Verificar como o projeto atual acessa email de usuarios (ex: no fluxo de convite original em `aceitar-convite`).

### RLS policy ue_insert_dono
Criada em 7.1 com `WITH CHECK (empresa_id = fn_get_empresa_id() AND fn_get_user_role() = 'dono')`. Para admin tambem poder convidar (motorista), a policy precisa ser: `fn_get_user_role() IN ('dono', 'admin')`. Verificar com a migration de 7.1 — se a policy so permite dono, ajustar nesta story.

### Validacao de role no server action
A RLS protege o INSERT, mas a validacao de qual role pode ser atribuido deve estar tambem no server action (defense in depth):
- Se `fn_get_user_role() = 'admin'` e `role = 'admin'`: rejeitar com erro
- Se `fn_get_user_role() = 'motorista'`: rejeitar (nao deveria chegar aqui pela UI, mas validar)

### Fluxo de convite existente
O sistema ja tem um fluxo de `aceitar-convite` para novos usuarios. Esta story e para usuarios que JA existem no sistema. O fluxo de email para novos usuarios e escopo futuro (AC: 5 — apenas orientar).

### Dependencias

### Testing

- Admin tenta convidar como admin: deve ser rejeitado
- Motorista nao ve botao de convite
