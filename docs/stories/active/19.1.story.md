## Story
**As a** dono de frota no plano profissional ou superior,
**I want** ver no BI um card que compara o periodo atual com o anterior (receita, gastos, lucro, margem, viagens, km/L) com setas de variacao,
**so that** eu saiba na hora se meu resultado ta melhorando ou piorando, sem precisar perguntar pro Copilot.

## Context

O Copilot ja tem a tool `comparativo_temporal` (T12) que faz essa comparacao via chat. Porem o Copilot consome tokens LLM (custo por request), enquanto o BI e zero custo (SQL direto). Features que sao "consulta passiva" (o dono so olha) devem viver no BI. O Copilot complementa quando o dono quer aprofundar ("por que o gasto subiu?").

### Gating por plano

O FrotaViva tem 4 planos: `free | essencial | profissional | enterprise`.

| Feature BI | free | essencial | profissional | enterprise |
|-----------|------|-----------|--------------|------------|
| KPI cards (lucro, margem, receita, custo) | sim | sim | sim | sim |
| Alertas de anomalia | sim | sim | sim | sim |
| Tendencia mensal (barras) | sim | sim | sim | sim |
| **Card comparativo temporal** | nao | nao | **sim** | **sim** |
| Benchmark setor | nao | nao | **sim** | **sim** |
| Simulador de viagem | nao | nao | **sim** | **sim** |

**Motivo:** o comparativo temporal e uma feature de analise avancada. No plano free/essencial, o dono ja ve os KPIs do periodo. O comparativo mostra evolucao, o que gera mais valor e justifica o upgrade.

### Utility de gating

Hoje nao existe mecanismo de feature gating por plano no BI. Esta story deve criar uma utility reutilizavel (`canAccessFeature(plano, feature)`) que sera usada por esta e futuras features gated.

## Technical Notes

- O campo `plano` ja existe na tabela `empresa` (tipo enum `plano_tipo`)
- `BiKpiCards` ja calcula receita/gasto/lucro/margem para o periodo selecionado
- O periodo anterior deve ser calculado automaticamente: se filtro = 30 dias, anterior = 30 dias antes disso
- Valores monetarios em centavos (CON-9), converter pra reais na exibicao
- km/L vem da tabela `gasto` (campos `km` e `litros`)
- A query pode reutilizar logica similar a `getBIKpis` mas com dois ranges de data

### Modelo de seguranca do gating (NON-NEGOTIABLE)

```
Server Action (getBIComparativo)
  1. Ler plano da empresa do usuario autenticado
  2. canAccessFeature(plano, 'bi_comparativo') === false?
     → return { data: null, locked: true, requiredPlan: 'profissional' }
     → NENHUMA query executada, NENHUM dado retornado
  3. Se pode acessar:
     → executar queries normalmente
     → return { data: { atual, anterior, variacao }, locked: false }

Client Component (BiComparativo)
  - Recebe props.locked === true? → renderiza <BiPlanGate />
  - Recebe props.data? → renderiza card comparativo normal
  - NUNCA recebe dados reais quando locked
```

**Por que nao blur/overlay:**
- CSS blur pode ser removido via DevTools (Inspect → delete style)
- opacity:0 pode ser mudado pra opacity:1
- display:none pode ser removido
- O HTML com dados reais ficaria no DOM acessivel
- A unica protecao real e NAO enviar o dado

## Out of Scope

- Comparativo por caminhao individual (futuro)
- Comparativo por motorista individual (futuro)
- Grafico de tendencia com mais de 2 periodos (ja existe em BiTendenciaMensal)
- Alterar plano do usuario (fluxo de billing/upgrade)
- Gating no Copilot (Copilot tem rate limit proprio)
