# Story 21.5: RLS combustivel_preco com role check — separar SELECT de INSERT/UPDATE/DELETE

## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que as policies RLS da tabela `combustivel_preco` diferenciem SELECT (disponivel para motoristas) de INSERT/UPDATE/DELETE (restrito a dono e admin),
**so that** motoristas nao possam criar, alterar ou deletar precos de combustivel da empresa.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Migration nova remove a policy atual de `combustivel_preco` (se existir policy ALL ou generalizada) e cria policies separadas.
2. Policy SELECT: `empresa_id = fn_get_empresa_id()` — qualquer usuario autenticado da empresa pode ler precos.
3. Policy INSERT: `empresa_id = fn_get_empresa_id() AND fn_get_user_role() IN ('dono', 'admin')` — somente dono e admin inserem.
4. Policy UPDATE: mesma condicao do INSERT (AC: 3).
5. Policy DELETE: mesma condicao do INSERT (AC: 3).
6. Teste RLS negativo com role motorista: usuario com role `motorista` nao consegue INSERT, UPDATE ou DELETE em `combustivel_preco`.
7. Teste RLS positivo com role dono: usuario com role `dono` consegue INSERT, UPDATE e DELETE.
8. Teste RLS positivo de SELECT com role motorista: usuario motorista consegue ler precos da propria empresa.
9. `supabase db push` aplica a migration sem erros em ambiente local.
10. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #105 (H4 data-engineer). Prioridade P1.

**Problema raiz:** Se a tabela `combustivel_preco` tem uma policy RLS do tipo ALL com condicao apenas `empresa_id = fn_get_empresa_id()`, qualquer usuario autenticado da empresa (incluindo motoristas) pode INSERT, UPDATE e DELETE em precos de combustivel. Isso e privilege escalation: motorista nao deveria alterar precos.

**Funcao `fn_get_user_role()`:** deve retornar o role do usuario autenticado (`dono`, `admin`, `motorista`, `gestor`). Verificar se essa funcao ja existe no banco. Se nao existir, criar como parte desta story.

**Estrutura de policies esperada:**

```sql
-- Remover policy existente (se for ALL)
DROP POLICY IF EXISTS "combustivel_preco_empresa" ON combustivel_preco;

-- SELECT: todos da empresa
CREATE POLICY "combustivel_preco_select"
ON combustivel_preco FOR SELECT
TO authenticated
USING (empresa_id = fn_get_empresa_id());

-- INSERT: apenas dono/admin
CREATE POLICY "combustivel_preco_insert"
ON combustivel_preco FOR INSERT
TO authenticated
WITH CHECK (
  empresa_id = fn_get_empresa_id()
  AND fn_get_user_role() IN ('dono', 'admin')
);

-- UPDATE: apenas dono/admin
CREATE POLICY "combustivel_preco_update"
ON combustivel_preco FOR UPDATE
TO authenticated
USING (empresa_id = fn_get_empresa_id())
WITH CHECK (
  empresa_id = fn_get_empresa_id()
  AND fn_get_user_role() IN ('dono', 'admin')
);

-- DELETE: apenas dono/admin
CREATE POLICY "combustivel_preco_delete"
ON combustivel_preco FOR DELETE
TO authenticated
USING (
  empresa_id = fn_get_empresa_id()
  AND fn_get_user_role() IN ('dono', 'admin')
);
```

**Verificar policies existentes:** inspecionar `supabase/migrations/` para entender o estado atual das policies de `combustivel_preco` antes de criar a migration de correcao.

**Impacto no codigo TypeScript:** nenhum — a restricao e no banco. Erro de permissao sera tratado pelo handler de erro existente nas server actions.
