## Story
**Como** dono do FrotaViva operando o piloto real,
**Quero** que erros em server actions criticas (fechamentos, viagens, upload, acertos) sejam capturados pelo Sentry com contexto de empresa e usuario, e que variaveis de ambiente criticas sejam validadas no startup,
**Para que** eu saiba rapidamente se algo esta falhando silenciosamente em producao antes que o usuario do piloto me diga.

## Acceptance Criteria

1. Um modulo `lib/observability/logger.ts` exporta funcao `logError(context: { action: string; empresaId?: string; usuarioId?: string }, error: unknown): void` que chama `Sentry.captureException` com as tags `action`, `empresa_id`, `usuario_id`.
2. Todos os blocos `catch` em server actions de fechamentos, viagens, gastos e upload de comprovante usam `logError()` em vez de (ou alem de) `console.error`.
3. O `logError` adiciona breadcrumbs de contexto: nome da action, parametros nao-sensiveis (ex: `motorista_id` sim, `senha` nao), timestamp.
4. Pelo menos 10 actions criticas sao instrumentadas: `criarViagem`, `editarViagem`, `excluirViagem`, `criarFechamento`, `pagarFechamento`, `criarGasto`, `uploadComprovante`, `criarMotorista`, `alterarSenha`, `switchEmpresa`.
5. Um modulo `lib/env.ts` exporta objeto validado com Zod contendo todas as env vars obrigatorias: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `SENTRY_DSN`, `NEXT_PUBLIC_SENTRY_DSN`. Se alguma estiver faltando, o app falha LOUD no startup com mensagem clara.
6. `instrumentation.ts` (ou `app/layout.tsx` server-side) importa `lib/env.ts` para validar no boot.
7. `npm run typecheck && npm run lint && npm run build` passam.
8. Em dev local, remover `SENTRY_DSN` do `.env.local` faz o startup falhar com erro descritivo (nao com "undefined is not a function").
9. Em producao, um erro proposital em uma server action instrumentada aparece no dashboard do Sentry em menos de 30 segundos.

## Dev Notes

- Sentry ja esta integrado no FrotaViva (confirmed pela auditoria @qa - Lighthouse mostra Sentry). O que falta e a instrumentacao estruturada nas actions.
- NAO loggar dados sensiveis: senha, tokens, valores financeiros em texto. `logError` deve sanitizar automaticamente campos conhecidos.
- O Zod schema de env vars pode crescer. Comecar com as 5 criticas listadas. Adicionar novas conforme o app evolui.
- O `instrumentation.ts` e feature nativa do Next.js 14+. Usar se disponivel; fallback para server layout se nao.
- Esta story nao depende de nenhuma outra alem de 14.1 (CI verde). Pode rodar no Sprint 3 em paralelo com Epic C e D.
- Arquivos previsivelmente criados: `lib/observability/logger.ts`, `lib/env.ts`. Afetados: 10+ server actions em `app/(dashboard)/**/actions.ts`, `instrumentation.ts`.
