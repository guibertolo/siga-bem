## Story
**Como** desenvolvedor do FrotaViva,
**Quero** que o ESLint bloqueie automaticamente o uso de classes Tailwind com cores nativas (`amber-*`, `slate-*`, `gray-*`, `red-*`, etc) em componentes,
**Para que** nenhuma regressao visual entre light e dark mode consiga chegar em producao sem ser explicitamente aprovada.

## Acceptance Criteria

1. Uma regra ESLint custom (ou uso do plugin `eslint-plugin-tailwindcss` com config custom, o que for mais pratico) bloqueia `className` contendo a regex `\b(amber|slate|gray|red|blue|green|yellow|orange|pink|purple|indigo|teal|cyan|rose|lime|emerald|sky|violet|fuchsia|zinc|neutral|stone)-[0-9]+` fora de `app/globals.css`.
2. A regra e configurada em `.eslintrc` (ou equivalente) e ativa por padrao em todos os arquivos `.tsx` em `components/` e `app/`.
3. A regra permite o uso dessas classes dentro de comentarios ou strings de documentacao (criterio de falso positivo razoavel).
4. A regra emite mensagem de erro clara em PT-BR: "Classe de cor nativa do Tailwind detectada. Use token do design system (ex: text-c-primary-600, bg-c-badge-warning-bg). Ver docs/design-system/tokens.md."
5. `npm run lint` executado logo apos a story 11.1 passa (nenhuma ocorrencia de cores nativas sobrou).
6. Uma tentativa deliberada de reintroduzir `className="bg-amber-100"` em um arquivo de teste produz erro de lint.
7. O hook `pre-commit` (story 14.1) bloqueia o commit quando a regra falha.

## Tasks / Subtasks

## Dev Notes

- Dependencia: story 11.1 precisa estar completa antes da 11.2 entrar em merge. Caso contrario o lint quebra no primeiro commit pos-merge.
- Ordem de execucao ideal dentro do sprint: 11.1 primeiro (trabalho manual), 11.2 depois (ativa a barreira).
- Considerar adicionar lista de exceptions na config (ex: `app/globals.css`) para nao gerar falso positivo na definicao dos tokens.
- NAO forcar que textos em emails transacionais (se houver) ou em paginas de status especiais obedecam. Se houver caso de uso legitimo, adicionar exception especifica, nao desabilitar a regra.

## Testing

- Unit tests: se a regra for custom em JavaScript puro, adicionar teste simples em `tests/eslint-rules/` que cria um source string com classe proibida e valida que a regra retorna erro.
- Integration tests: rodar `npm run lint` no codebase inteiro.
- Manual testing:
  - Criar `components/teste-lint.tsx` com `<div className="bg-amber-100" />`, rodar lint, ver erro.
  - Trocar para `<div className="bg-c-badge-warning-bg" />`, rodar lint, ver sucesso.
  - Deletar o arquivo de teste.
