## Story
**Como** desenvolvedor do FrotaViva,
**Quero** ter tipos TypeScript gerados automaticamente a partir do schema do Supabase e um unico tipo `Usuario` canonico,
**Para que** o codigo pare de usar 62 casts `as unknown as`, ganhe seguranca de tipos real e elimine divergencias entre definicoes duplicadas de usuario.

## Acceptance Criteria

1. Um script `npm run db:types` existe em `package.json` e executa `npx supabase gen types typescript --local > types/database.ts` (ou contra o projeto remoto se local nao estiver disponivel).
2. O arquivo `types/database.ts` e gerado e commitado na primeira execucao. Ele contem tipos derivados de todas as tabelas, views, functions e enums do schema publico.
3. O tipo `Usuario` existe em um unico lugar (`types/usuario.ts` ou equivalente) e e importado de todas as camadas que precisam dele. Qualquer duplicata de interface `Usuario` e removida.
4. Server actions e queries que hoje usam `as unknown as` para contornar tipos do Supabase passam a usar os tipos gerados. Pelo menos 50 dos 62 casts existentes sao substituidos.
5. `npm run typecheck` passa sem novos erros apos a substituicao.
6. Um helper utilitario `singleRelation<T>(value: T | T[]): T | null` e criado em `lib/utils/supabase-types.ts` para lidar com relations que o Supabase retorna como array mas o dominio trata como objeto unico.
7. README de desenvolvimento (ou `docs/architecture/database-types.md`) documenta como regenerar tipos apos migration nova.
8. CI nao quebra por conta da mudanca. O build passa.
