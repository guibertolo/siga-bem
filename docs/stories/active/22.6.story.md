## Story
**As a** dono de frota,
**I want** que o export CSV de historico financeiro valide os IDs de motoristas contra minha empresa antes de processar,
**so that** uma URL forjada com IDs de outra empresa nao retorne dados confidenciais dessa empresa.

## Epic
Epic 22 — Infra Hardening.

## Origem
Addendum 2026-04-14, item #6

## Acceptance Criteria

1. O endpoint `app/(dashboard)/financeiro/historico/export/route.ts` valida `motoristaIds` recebidos na URL/body **antes** de executar o `.in('motorista_id', motoristaIds)` no banco.
2. A validacao consiste em: buscar quais dos `motoristaIds` informados pertencem a alguma empresa do usuario autenticado (via `usuario_empresa` + `motorista.empresa_id`). Apenas IDs validos sao passados para o `.in()`.
3. Se nenhum `motoristaId` valido for encontrado (todos eram de outra empresa ou lista vazia), o endpoint retorna CSV vazio (zero linhas de dados, apenas header) com status 200, sem retornar erro que exponha a existencia dos IDs.
4. A validacao ocorre mesmo que a consulta subsequente seja protegida por RLS — a defesa em profundidade garante que RLS e a segunda linha, nao a unica.
5. O comportamento de export para o usuario legitimo (com IDs proprios) e identico ao atual.
6. `npm run lint && typecheck && build` passam.
7. Teste: usuario A envia `motoristaIds` contendo IDs de motoristas da empresa do usuario B — resposta deve ser CSV vazio, sem dados de B.

## Technical Notes

- Arquivo a modificar: `app/(dashboard)/financeiro/historico/export/route.ts`.
- Fluxo de validacao sugerido:
  1. Obter empresas do usuario autenticado via `usuario_empresa`.
  2. Buscar `motorista` onde `id IN (motoristaIds)` E `empresa_id IN (empresas_do_usuario)`.
  3. Usar apenas os IDs retornados nessa query para o `.in()` final.
- Este padrao e "validacao de ownership antes da query de dados" — simples e auditavel.
- Nao e necessario alterar a assinatura da rota ou o formato do CSV.
- Complexidade: M / Risco S (adicionar query de validacao antes da existente).

## Testing Requirements

- Teste de seguranca: usuario A (empresa X) envia `motoristaIds` de motoristas da empresa Y — espera CSV com apenas header.
- Teste de regressao: usuario A envia seus proprios `motoristaIds` — espera CSV identico ao comportamento atual.
- Teste edge case: lista de `motoristaIds` vazia — espera CSV com apenas header.
