## Story
**Como** desenvolvedor do FrotaViva,
**Quero** que o middleware de autenticacao proteja todas as rotas por padrao (whitelist inversa) e que o layout do dashboard nao chame `getUser()` duas vezes,
**Para que** a protecao de rotas seja segura por default e o custo de cada carregamento do dashboard caia uma round trip ao Supabase.

## Acceptance Criteria

1. O `middleware.ts` (ou equivalente) passa a proteger TODAS as rotas por default. Uma whitelist explicita lista as rotas publicas (`/`, `/login`, `/signup`, `/auth/callback`, `/api/public/*`, assets estaticos).
2. Qualquer rota NAO listada na whitelist exige usuario autenticado. Rotas nao autenticadas redirecionam para `/login`.
3. O layout do dashboard (`app/(dashboard)/layout.tsx`) chama `getUser()` UMA unica vez. Qualquer chamada duplicada e removida.
4. Se uma camada filha (page ou server component) precisa do usuario, ele e passado via props ou context - nao re-fetched.
5. `npm run typecheck && npm run lint && npm run build` passam.
6. Fluxo manual: acessar `/dashboard` sem login -> redireciona pra `/login`. Fazer login -> acessa dashboard sem erro. Acessar `/` sem login -> landing page publica renderiza normal.
7. Tentar acessar rota nao listada em whitelist como `/abracadabra-xyz` sem login -> middleware responde com redirect pra `/login`, nao com 404 publico.
8. Medicao simples (adicionar log temporario ou usar devtools network) confirma que `/dashboard` faz 1 chamada a Supabase de `getUser`, nao 2.
