## Story
**Como** desenvolvedor do FrotaViva,
**Quero** uma suite de testes de integracao que valide RLS em todas as tabelas criticas do Supabase,
**Para que** qualquer alteracao futura em policies ou tabelas detecte imediatamente se um usuario de empresa A pode acessar dados de empresa B.

## Acceptance Criteria

1. A suite roda contra Supabase local (`supabase start` + `supabase db push`). Docker e pre-requisito de dev.
2. Setup de teste cria 2 empresas, 2 donos (um por empresa), 2 motoristas (um por empresa). Dados de seed para teste.
3. Testes negativos de cross-tenant para pelo menos estas tabelas: `viagem`, `gasto`, `fechamento`, `fechamento_item`, `motorista`, `caminhao`, `usuario_empresa`.
4. Cada teste negativo valida: autenticando como dono A, tentar SELECT em registros de empresa B retorna 0 linhas (nao erro, apenas vazio por RLS).
5. Cada teste negativo valida: autenticando como dono A, tentar INSERT de viagem em empresa B falha (RLS INSERT policy).
6. Cada teste negativo valida: autenticando como motorista A, tentar SELECT de viagens de motorista B retorna 0 linhas.
7. Testes positivos confirmam: dono A ve todos os registros de empresa A. Motorista A ve registros proprios de empresa A.
8. Suite roda no CI como step separado. Se falhar, CI fica vermelho.
9. Suite completa em menos de 60 segundos (Supabase local e rapido; se demorar mais, otimizar setup).
10. Cada teste tem nome descritivo em PT-BR no `describe`/`it`, ex: `it('dono A nao ve viagens de empresa B')`.

## Dev Notes

- Supabase local exige Docker. Se CI nao tiver Docker, considerar `supabase link` com projeto de testes dedicado (mais lento, menos ideal, mas funciona).
- Alternativa: usar GitHub Actions com servico Docker. Supabase disponibiliza `supabase/postgres` como container.
- Setup pode usar `supabase.auth.signUp` com email/senha para criar usuarios de teste, depois autenticar como cada um.
- Funcoes RPC como `fn_get_user_empresas` precisam ser testadas tambem: retornam apenas empresas do usuario logado.
- NAO testar RLS em tabelas que nao tem policies (ex: se `auth.users` nao tem RLS explicita, nao testar).
- Esta story e dependencia de 10.3 (refactor) e 13.x (acertos v2). Deve sair cedo no sprint.
- Arquivos previsivelmente criados: `tests/integration/rls/`.
