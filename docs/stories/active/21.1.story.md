## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que a view `view_viagens_ativas` rode com `security_invoker = true`,
**so that** as RLS policies do usuario chamador sejam respeitadas e nenhum dado cross-tenant vaze via essa view.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Migration nova recria a view `view_viagens_ativas` com `WITH (security_invoker = true)`.
2. A view nao expoe nenhuma coluna ou linha de empresa diferente da empresa do usuario autenticado.
3. Teste RLS cross-tenant negativo: usuario autenticado como `dono_empresa_A` ao consultar a view nao recebe nenhuma linha pertencente a `empresa_B`.
4. Teste RLS positivo: usuario autenticado como `dono_empresa_A` recebe apenas as viagens ativas de `empresa_A`.
5. `supabase db push` aplica a migration sem erros em ambiente local.
6. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #102 (H1 data-engineer). Prioridade P0.

**Problema raiz:** Views sem `security_invoker` rodam como owner (postgres/superuser), ignorando as RLS policies do usuario chamador. Se um atacante ou bug fizer uma query anonima/incorreta contra a view, todos os dados de todas as empresas ficam expostos.

**Correcao padrao Supabase:** `ALTER VIEW view_viagens_ativas SET (security_invoker = true);` OU dropar e recriar com `CREATE OR REPLACE VIEW view_viagens_ativas WITH (security_invoker = true) AS ...`.

**Localizacao da view:** procurar em `supabase/migrations/` o arquivo que criou `view_viagens_ativas`. Mapear a query SELECT atual antes de recriar.

**Estrategia de migration:**
1. Localizar o SELECT original da view.
2. Criar migration: `DROP VIEW IF EXISTS view_viagens_ativas; CREATE VIEW view_viagens_ativas WITH (security_invoker = true) AS <mesmo SELECT>;`.
3. Garantir que as RLS policies em `viagem` (ou tabela base da view) filtrem por `empresa_id = fn_get_empresa_id()`.

**Tabelas base provaveis:** `viagem`, `motorista`, `caminhao` — todas possuem RLS com filtro de empresa. Com `security_invoker`, a view herda essas policies automaticamente.

**Nao e necessario** modificar o codigo TypeScript: a view continuara sendo chamada com o mesmo nome e interface.
