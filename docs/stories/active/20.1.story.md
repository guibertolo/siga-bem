## Story
**As a** dono de frota,
**I want** que o odometro (km de saida e km de chegada) seja obrigatorio em toda viagem,
**so that** todos os calculos de km/L, R$/km, CPK e rentabilidade por rota sejam confiaveis, incluindo os km rodados entre viagens (gaps/deslocamentos vazios).

## Context

Hoje `km_saida` e `km_chegada` sao opcionais. Quando o dono nao preenche, metade das metricas do Copilot e do BI ficam sem sentido (km/L, R$/km, CPK). Alem disso, os km entre viagens (gaps) nao sao contabilizados em nenhum lugar, o que subestima o km total real do caminhao.

### O problema dos gaps

```
Caminhao ABC-1234 (mes de abril)

Viagem 1: km_saida 50.000 → km_chegada 50.800  = 800 km (trip)
  [gap: 50.800 → 51.100 = 300 km vazio — deslocamento, reposicionamento]
Viagem 2: km_saida 51.100 → km_chegada 52.000  = 900 km (trip)
  [gap: 52.000 → 52.050 = 50 km vazio]
Viagem 3: km_saida 52.050 → km_chegada 52.900  = 850 km (trip)

Total trip km:  2.550
Total gap km:     350
Total real km:  2.900 (odometro final - odometro inicial)
Taxa de vazio:   12%
```

Sem os gaps, o km/L parece pior do que e na realidade (mesmo combustivel, menos km contabilizados).

### Onde cada metrica usa o que

| Metrica | Trip km | Gap km | Km total (real) |
|---------|---------|--------|-----------------|
| Margem por viagem | sim | nao | nao |
| Frete/km por viagem | sim | nao | nao |
| km/L do caminhao | nao | nao | **sim** |
| CPK do caminhao | nao | nao | **sim** |
| R$/km da frota | nao | nao | **sim** |
| Taxa de vazio | nao | sim | sim |
| Rentabilidade por rota | sim | nao | nao |

## Technical Notes

### Campos no banco (ja existem)

```sql
-- viagem (ja existe)
km_saida    integer NULL  -- sera NOT NULL para novas viagens via validacao app-level
km_chegada  integer NULL  -- sera NOT NULL ao concluir via validacao app-level

-- caminhao (ja existe)
km_atual    integer NOT NULL DEFAULT 0  -- atualizado com km_chegada ao concluir
```

**NAO alterar constraint do banco para NOT NULL** — quebraria viagens historicas. A obrigatoriedade e enforced via validacao no Zod schema e server actions (app-level).

### Calculo de gap (pseudocodigo)

```typescript
// Viagens do caminhao ABC-1234 em ordem cronologica
const viagens = [
  { km_saida: 50000, km_chegada: 50800 }, // trip 1
  { km_saida: 51100, km_chegada: 52000 }, // trip 2
  { km_saida: 52050, km_chegada: 52900 }, // trip 3
];

const gaps = [];
for (let i = 0; i < viagens.length - 1; i++) {
  const gap = viagens[i + 1].km_saida - viagens[i].km_chegada;
  gaps.push(Math.max(0, gap)); // clamp negativo pra 0
}
// gaps = [300, 50]

const tripKm = viagens.reduce((sum, v) => sum + (v.km_chegada - v.km_saida), 0);
const gapKm = gaps.reduce((sum, g) => sum + g, 0);
const totalKm = tripKm + gapKm;
const taxaVazio = totalKm > 0 ? (gapKm / totalKm) * 100 : 0;
```

### Pre-preenchimento inteligente

Ao abrir o form de nova viagem:
1. Buscar `caminhao.km_atual` do caminhao selecionado
2. Se > 0, pre-preencher `km_saida` com esse valor
3. O dono confirma ou ajusta (odometro real pode ser ligeiramente diferente)

Ao concluir viagem:
1. Buscar `caminhao.km_atual` atual
2. Se > `viagem.km_saida`, pre-preencher `km_chegada`
3. O dono confirma ou ajusta

### Threshold de warning

Gap > 2.000 km sem viagem registrada gera warning. Esse threshold pode ser ajustado. Nao e bloqueante porque:
- O caminhao pode ter ficado parado meses e depois viajado longe
- O dono pode ter feito viagens sem registrar (catch-up de dados)
- Erro de digitacao no odometro

- GPS/telemetria automatica (muito futuro)
