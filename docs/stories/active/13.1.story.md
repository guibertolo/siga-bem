## Story
**Como** dono de cegonha usuario do FrotaViva,
**Quero** que o sistema suporte acertos individuais (avulsos) desvinculados de viagens especificas,
**Para que** eu possa lancar pagamentos pontuais como bonus, descontos ou adiantamentos sem ter que inventar uma viagem fake.

## Acceptance Criteria

1. Uma migration nova em `supabase/migrations/` cria (se ainda nao existir) um ENUM `fechamento_item_tipo` com valores `viagem`, `avulso`, `ajuste`. Se ja existir `viagem`, adiciona `avulso` e `ajuste` ao enum.
2. A coluna `fechamento_item.tipo` usa esse enum. Se antes era `text` com CHECK constraint, a migration muda o tipo e remove o CHECK antigo.
3. A coluna `fechamento_item.referencia_id` aceita NULL quando `tipo = 'avulso'` ou `tipo = 'ajuste'`. Para `tipo = 'viagem'`, continua obrigatoria (CHECK constraint).
4. Uma nova coluna `fechamento_item.descricao TEXT` existe para comportar a descricao do acerto avulso (NULL quando tipo='viagem').
5. O UNIQUE INDEX `uq_fi_viagem_unica` (criado na story FIRE) e ajustado para `WHERE tipo = 'viagem'`. Acertos avulsos nao sofrem restricao de unicidade por referencia_id.
6. Rollback script inverso existe no mesmo arquivo (comentado ou em secao separada) documentando como reverter a migration.
7. A migration e aplicada com sucesso em ambiente local via `supabase db push` em uma base limpa.
8. A migration e aplicada com sucesso em uma base clonada de producao (clone) sem perda de dados. Testado antes de liberar para producao real.
9. Tipos Supabase regenerados (via story 10.1) refletem o novo enum e a nova coluna.

## Dev Notes

- Pre-requisito: a migration FIRE do @devops (ver `docs/review/2026-04-12/fire-block-execution.md`) ja criou o UNIQUE INDEX parcial. Esta story AJUSTA esse index, nao cria do zero.
- Pre-requisito: story 14.3 (testes de fechamentos) deve estar no ar. Mudar schema de fechamento sem testes e risco financeiro inaceitavel.
- Cuidado com enums no Postgres: adicionar valor em enum nao e transacional por padrao. Usar `ALTER TYPE ... ADD VALUE IF NOT EXISTS ...` e entender que DDL de enum pode exigir commit antes de usar o novo valor.
- `descricao` com limite razoavel via CHECK (ex: `char_length(descricao) <= 200`) evita texto muito longo.
- Esta story cuida SO do schema. A server action `criarAcertoAvulso` e a UX vao na story 13.2.
- Depois desta story, o campo `referencia_id` pode ser NULL, entao qualquer query que fizer `.referencia_id` direto sem check vai quebrar. A refatoracao dessas queries e parte da story 13.2.

## Testing

- Unit tests: n/a.
- Integration tests: rodar suite de fechamentos existente (story 14.3). Todos devem passar.
- Manual testing:
  - Apos aplicar migration, tentar inserir via SQL: `INSERT INTO fechamento_item (fechamento_id, tipo, valor, descricao) VALUES (...)` com `tipo = 'avulso'` e `referencia_id = NULL`. Deve funcionar.
  - Tentar inserir `tipo = 'viagem'` com `referencia_id = NULL`. Deve falhar (CHECK constraint).
  - Tentar inserir duas viagens com mesmo `referencia_id`. Deve falhar (UNIQUE INDEX).
  - Tentar inserir dois acertos avulsos sem referencia. Deve funcionar.
