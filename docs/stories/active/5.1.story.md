## Story
**As a** desenvolvedor do Siga Bem,
**I want** criar a tabela `abastecimento_detalhe` com RLS, indices, views e funcoes de BI no banco Supabase e atualizar os types TypeScript correspondentes,
**so that** as stories seguintes possam persistir dados de abastecimento (litros, preco/litro, posto, cidade/UF) vinculados a gastos de combustivel sem duplicar a infraestrutura financeira ja existente.

## Acceptance Criteria

1. Migration SQL executada com sucesso em ambiente de desenvolvimento: cria a tabela `abastecimento_detalhe` com todos os campos, constraints e trigger de `updated_at`.
2. Tabela `abastecimento_detalhe` possui relacao 1:1 com `gasto` via `gasto_id UUID NOT NULL UNIQUE REFERENCES gasto(id) ON DELETE CASCADE`.
3. Trigger `trg_validate_abastecimento_gasto` impede INSERT/UPDATE em `abastecimento_detalhe` quando o `gasto` referenciado nao for da categoria "Combustivel".
4. RLS habilitada na tabela com 4 policies (select, insert, update, delete) usando o padrao existente `fn_get_empresa_id()` e `fn_get_motorista_id()`.
5. Indices criados: `idx_abdet_cidade_estado (estado, cidade)`, `idx_abdet_tipo_combustivel (tipo_combustivel)`, e `idx_gasto_viagem ON gasto(viagem_id) WHERE viagem_id IS NOT NULL`.
6. View `vw_media_combustivel_regiao` criada com campos: `empresa_id, estado, cidade, tipo_combustivel, total_abastecimentos, preco_medio_litro, preco_min_litro, preco_max_litro, total_litros, total_centavos, primeira_data, ultima_data`.
7. View `vw_custo_combustivel_caminhao` criada com campos: `empresa_id, caminhao_id, placa, modelo, tipo_combustivel, total_abastecimentos, total_centavos, total_litros, preco_medio_litro, km_por_litro_estimado, primeiro_abastecimento, ultimo_abastecimento`.
8. View `vw_custo_combustivel_motorista` criada com campos analogo ao caminhao mas agrupado por motorista.
9. View `vw_custo_combustivel_viagem` criada com campos: `empresa_id, viagem_id, origem, destino, km_estimado, motorista_id, motorista_nome, total_abastecimentos, total_centavos, total_litros, preco_medio_litro, km_por_litro_viagem`.
10. Funcao `fn_estimativa_custo_viagem(p_empresa_id, p_km_estimado, p_consumo_medio_km_l, p_tipo_combustivel)` criada com fallback seguro para consumo padrao de 2.5 km/l quando nao houver historico.
11. Script de rollback documentado e funcional (DROP TABLE, DROP VIEWs, DROP FUNCTIONs em ordem reversa).
12. Types TypeScript atualizados: interface `AbastecimentoDetalhe` em `types/abastecimento.ts` com todos os campos da tabela, e `AbastecimentoDetalheInsert` / `AbastecimentoDetalheUpdate` correspondentes.
13. Arquivo `types/database.ts` atualizado com a nova tabela no schema de tipos gerado pelo Supabase (ou interface manual compativel).
14. Gastos existentes sem `abastecimento_detalhe` continuam funcionando sem erros (backward compatible).

## Dev Notes

### Decisao Arquitetural
A abordagem escolhida e **Opcao Hibrida** (detalhada em `docs/database/schema-abastecimento.md`): o registro financeiro continua em `gasto` (categoria "Combustivel"), e os dados tecnicos do abastecimento (litros, preco/litro, posto, localizacao) ficam em `abastecimento_detalhe` com FK 1:1 para `gasto`.

**Nao** adicionar colunas nullable em `gasto`. O documento `docs/architecture-combustivel-notas.md` propoe estender `gasto`, mas o schema definitivo esta em `docs/database/schema-abastecimento.md` (mais recente e detalhado).

### Schema da Tabela
```
abastecimento_detalhe
  id                UUID PK DEFAULT gen_random_uuid()
  gasto_id          UUID NOT NULL UNIQUE REFERENCES gasto(id) ON DELETE CASCADE
  litros            NUMERIC(10,2) NOT NULL  -- litros com 2 casas decimais
  preco_litro       NUMERIC(10,4) NOT NULL  -- R$/litro com 4 casas decimais (ex: 6.4590)
  tipo_combustivel  combustivel_tipo NOT NULL DEFAULT 'diesel_s10'
  km_odometro       INTEGER                 -- leitura do odometro, opcional
  posto             TEXT                    -- nome do posto, texto livre
  bandeira          TEXT                    -- Shell, BR, Ipiranga, etc.
  cidade            TEXT NOT NULL
  estado            CHAR(2) NOT NULL
  observacao        TEXT
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()

  CONSTRAINT ck_abdet_litros  CHECK (litros > 0)
  CONSTRAINT ck_abdet_preco   CHECK (preco_litro > 0)
  CONSTRAINT ck_abdet_km      CHECK (km_odometro IS NULL OR km_odometro >= 0)
  CONSTRAINT ck_abdet_estado  CHECK (estado ~ '^[A-Z]{2}$')
```

### ENUM existente
O tipo `combustivel_tipo` (`diesel_s10`, `diesel_comum`) ja existe no banco. Confirmar com `\dT+ combustivel_tipo` antes de referenciar.

### Funcoes auxiliares existentes
- `fn_set_updated_at()` — trigger function para auto-update de `updated_at`, ja existe no banco
- `fn_get_empresa_id()` — retorna empresa_id do usuario autenticado, usada em RLS
- `fn_get_motorista_id()` — retorna motorista_id do usuario autenticado, usada em RLS
- `fn_get_user_role()` — retorna role do usuario: `'dono'`, `'admin'` ou `'motorista'`

### Categoria Combustivel
O seed ja inclui `categoria_gasto` com `nome = 'Combustivel'`. O trigger de validacao faz JOIN com `categoria_gasto` para verificar o nome. Nao usar UUID fixo — fazer query pelo nome.

### Fluxo de dados completo
```
1. INSERT gasto (empresa_id, categoria_id='Combustivel', valor, motorista_id, viagem_id, ...)
2. INSERT abastecimento_detalhe (gasto_id, litros, preco_litro, cidade, estado, ...)
3. INSERT foto_comprovante (gasto_id, storage_path, ...) — opcional
```
Tudo em transacao unica. O trigger em `abastecimento_detalhe` valida que o gasto e da categoria correta.
