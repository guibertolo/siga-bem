## Story
**As a** desenvolvedor do Siga Bem,
**I want** estender a tabela `gasto` com colunas de abastecimento (litros, tipo_combustivel, posto_local, uf_abastecimento) e criar indices parciais e views de BI para analytics,
**so that** as stories seguintes possam persistir e analisar dados reais de abastecimento sem criar uma tabela separada, mantendo toda a infraestrutura financeira existente (fechamento, RLS, foto_comprovante) funcionando sem alteracoes.

## Acceptance Criteria

1. Migration SQL executada com sucesso em ambiente de desenvolvimento: ALTER TABLE `gasto` adiciona as 4 colunas novas sem erros.
2. As 4 colunas adicionadas sao: `litros NUMERIC(10,3)`, `tipo_combustivel combustivel_tipo`, `posto_local TEXT`, `uf_abastecimento CHAR(2)`. Todas nullable para garantir backward compatibility com gastos existentes.
3. Constraint CHECK adicionada: `ck_gasto_litros CHECK (litros IS NULL OR litros > 0)`.
4. Constraint CHECK adicionada: `ck_gasto_uf CHECK (uf_abastecimento IS NULL OR uf_abastecimento ~ '^[A-Z]{2}$')`.
5. Indice parcial criado: `idx_gasto_combustivel ON gasto(empresa_id, data) WHERE tipo_combustivel IS NOT NULL` para otimizar queries de BI que filtram apenas gastos de combustivel.
6. Indice parcial criado: `idx_gasto_uf_combustivel ON gasto(uf_abastecimento, tipo_combustivel) WHERE uf_abastecimento IS NOT NULL` para otimizar queries de media por regiao.
7. View `vw_media_combustivel_regiao` criada com campos: `empresa_id, uf_abastecimento, tipo_combustivel, total_abastecimentos, preco_medio_litro, preco_min_litro, preco_max_litro, total_litros, total_valor_centavos, primeira_data, ultima_data`. O preco por litro e DERIVADO: `(valor::numeric / 100) / litros`.
8. View `vw_custo_por_caminhao` criada com campos: `empresa_id, caminhao_id, placa, modelo, total_abastecimentos, total_valor_centavos, total_litros, preco_medio_litro_derivado, km_por_litro_estimado, primeiro_abastecimento, ultimo_abastecimento`. Filtro `WHERE tipo_combustivel IS NOT NULL`.
9. View `vw_custo_por_motorista` criada com campos analogos ao caminhao mas agrupada por `motorista_id, motorista_nome`.
10. View `vw_gastos_bi` criada com TODOS os tipos de gasto (sem filtro de categoria): `empresa_id, categoria_nome, caminhao_id, motorista_id, viagem_id, valor, data, mes_ano`. Esta view suporta o BI completo da Story 5.5.
11. Script de rollback documentado e funcional: DROP INDEX, DROP VIEW, ALTER TABLE DROP COLUMN em ordem reversa.
12. Types TypeScript atualizados: interface `Gasto` em `types/gasto.ts` (ou equivalente) extendida com os 4 campos novos opcionais (`litros?: number | null`, `tipo_combustivel?: 'diesel_s10' | 'diesel_comum' | null`, `posto_local?: string | null`, `uf_abastecimento?: string | null`).
13. Gastos existentes sem os novos campos continuam funcionando sem erros — backward compatible (campos nullable nao quebram queries, forms ou components existentes).
14. O tipo `combustivel_tipo` ENUM (`diesel_s10`, `diesel_comum`) confirmado como existente no banco antes de referenciar. Se nao existir, criar na migration antes do ALTER TABLE.

## Dev Notes

### Decisao Arquitetural (correcao @po)
A abordagem e **Opcao B: estender tabela `gasto`** com campos nullable. NAO criar tabela separada `abastecimento_detalhe`. Razao:
- Reutiliza RLS existente sem novas policies
- `foto_comprovante` ja funciona com `gasto_id` — sem alteracoes
- `fn_calcular_fechamento` ja soma `gasto.valor` — combustivel entra automaticamente
- Formulario de gasto existente pode ser estendido (Story 5.2) sem transacao dupla

### Preco por litro e DERIVADO
O `preco_litro` NAO e armazenado. E calculado nas views e no frontend:
```sql
-- Em views SQL
(valor::numeric / 100.0) / litros AS preco_medio_litro_derivado

-- No frontend (TypeScript)
const precoLitro = gasto.litros && gasto.litros > 0
  ? (gasto.valor / 100) / gasto.litros
  : null
```

### NUMERIC(10,3) para litros
3 casas decimais — suficiente para cegonheiros (ex: 245.750 litros). NAO usar NUMERIC(10,2) que perde precisao em abastecimentos grandes.

### Indice parcial — performance BI
O indice `WHERE tipo_combustivel IS NOT NULL` garante que apenas registros de combustivel entram no indice. A maioria dos gastos nao e combustivel, entao o indice fica pequeno e rapido.

### Views sem RLS propria
As views herdam a RLS de `gasto`. O `security_invoker = true` garante que a RLS do usuario autenticado e aplicada ao acessar a view via Supabase client. Confirmar com:
```sql
CREATE OR REPLACE VIEW vw_media_combustivel_regiao
  WITH (security_invoker = true)
AS ...
```
