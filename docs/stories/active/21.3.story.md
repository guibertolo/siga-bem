## Story
**As a** administrador de seguranca do FrotaViva,
**I want** um helper `assertOwnership` centralizado e aplicado em todas as 14 mutations identificadas,
**so that** mesmo que uma falha de RLS ocorra no banco, uma camada de defesa-em-profundidade na aplicacao garanta que nenhum dado cross-tenant seja lido ou modificado.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Helper `assertOwnership(table, id, empresaId)` criado em `lib/security/assert-ownership.ts`.
2. O helper executa `SELECT id FROM {table} WHERE id = {id} AND empresa_id = {empresaId}` e lanca `SecurityError` se retornar vazio.
3. O helper usa o cliente Supabase autenticado do usuario (nao service_role).
4. Todas as 14 mutations listadas abaixo recebem `.eq('empresa_id', usuario.empresa_id)` adicionado a query de UPDATE/DELETE E uma chamada a `assertOwnership` antes da operacao principal.
5. Em caso de falha do `assertOwnership`, a mutation retorna `{ success: false, error: 'Acesso negado' }` sem indicar a existencia do recurso.
6. Testes unitarios do helper com mocks: caso positivo (recurso pertence a empresa), caso negativo (recurso de outra empresa), caso nao-encontrado.
7. `npm run lint && npm run typecheck && npm test` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #103 (H2 cyber-chief). Prioridade P1.

**Problema raiz:** As mutations usam o `id` recebido do frontend diretamente em UPDATE/DELETE sem verificar se o recurso pertence a empresa do usuario autenticado. Um atacante pode enumerar IDs e modificar recursos de outras empresas (IDOR — Insecure Direct Object Reference).

**Interface do helper:**

```typescript
// lib/security/assert-ownership.ts
export class SecurityError extends Error {
  constructor(message = 'Acesso negado') {
    super(message);
    this.name = 'SecurityError';
  }
}

export async function assertOwnership(
  supabase: SupabaseClient,
  table: string,
  id: string,
  empresaId: string
): Promise<void> {
  const { data, error } = await supabase
    .from(table)
    .select('id')
    .eq('id', id)
    .eq('empresa_id', empresaId)
    .single();

  if (error || !data) {
    throw new SecurityError();
  }
}
```

**Padrao de uso em server actions:**

```typescript
// Antes de qualquer UPDATE/DELETE:
const { empresaId } = await getUsuarioEmpresaId(); // funcao existente
await assertOwnership(supabase, 'viagem', viagemId, empresaId);

// Query de UPDATE deve incluir .eq('empresa_id', empresaId):
await supabase
  .from('viagem')
  .update({ ... })
  .eq('id', viagemId)
  .eq('empresa_id', empresaId); // defesa-em-profundidade
```

**`getMotorista` e `listComprovantes`:** sao operacoes de leitura mas tambem precisam de filtro por empresa. Usar `.eq('empresa_id', empresaId)` diretamente na query, sem necessariamente chamar `assertOwnership` (que e otimizado para UPDATEs/DELETEs — avaliar caso a caso).

**Localizacao do arquivo a criar:** `app/lib/security/assert-ownership.ts` (ajustar path conforme estrutura real do projeto).

**Tratamento de erro em Server Actions do Next.js:** lancamento de `SecurityError` deve ser capturado e retornar `{ error: 'Acesso negado' }` com status 403, sem stack trace para o cliente.
