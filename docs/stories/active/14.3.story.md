## Story
**Como** desenvolvedor do FrotaViva,
**Quero** testes de integracao cobrindo logica de fechamentos (transicoes de status, overlap de periodo, divisao por percentual, calculo de valor total) e funcoes puras de calculo,
**Para que** qualquer refatoracao ou feature nova em fechamentos (Epic B ou E) detecte regressao antes de chegar em producao com valores financeiros errados.

## Acceptance Criteria

1. Suite de teste para fechamentos cobre no minimo:
   a. Transicoes de status validas: `aberto` -> `fechado` -> `pago`. Transicoes invalidas rejeitadas (ex: `pago` -> `aberto` direto).
   b. Overlap de periodo: dois fechamentos no mesmo periodo para o mesmo motorista devem ser detectados.
   c. Divisao por percentual: se o motorista recebe 70%, o calculo do valor a pagar esta correto.
   d. Edge case: fechamento com zero viagens (deve ser permitido ou rejeitado, conforme regra de negocio existente).
   e. Edge case: viagem com valor zero (ex: repositionamento).
2. Teste unitario para `calcular_fechamento_total` (funcao pura): dado um array de `fechamento_item` com valores, calcula soma corretamente. Testa: lista vazia (= 0), lista com 1 item, lista com 5 itens, lista com decimais (precisao centavos).
3. Teste unitario para `aggregateKpis` (se existir como funcao pura): dado dados brutos, retorna KPIs agregados. Testa: dados de 1 empresa, dados de multi-empresa.
4. Snapshot test para o retorno de `getViagensPendentesAcerto`: dado um motorista com viagens em estados variados (pago, aberto, sem fechamento), o retorno contem exatamente as viagens esperadas. Snapshots sao versionados no repo.
5. Suite roda no CI junto com testes de RLS (story 14.2) ou como step separado.
6. Suite completa em menos de 30 segundos para funcoes puras e menos de 60s para integracao.
7. Nomes dos testes em PT-BR descritivo: `it('calcula total de fechamento com 3 viagens e percentual 70%')`.
