## Story
**As a** motorista,
**I want** tirar uma foto da chamada (documento com os VINs e o valor do frete) diretamente pelo celular ao criar a viagem,
**so that** o dono da frota tenha o documento digitalizado sem precisar me ligar ou esperar eu chegar.

## Epic
Epic 23 — REL (Relatorios e Upload da Chamada).

## Acceptance Criteria

1. Formulario de lancamento de viagem (`components/viagens/ViagemForm.tsx`) exibe botao "Tirar foto da chamada" com:
   - Altura minima de 64px, largura total do container
   - Icone de camera visivel
   - Input `type="file"` com `accept="image/jpeg,image/png,image/webp"` e `capture="environment"` (camera traseira como padrao no mobile, fallback para galeria em desktop)

2. Apos captura ou selecao, exibe tela de preview com:
   - Imagem em miniatura grande (minimo 200px de altura)
   - Botao "Usar esta foto" (cor verde, largura total)
   - Botao "Tirar de novo" (cor cinza, largura total)
   - Nao avanca para upload ate o usuario confirmar com "Usar esta foto"

3. Ao confirmar, compressao do lado do cliente antes do upload:
   - Usar `lib/utils/compress-image.ts` com novo parametro `outputType: 'image/webp'` (max 1600px, qualidade 0.8)
   - Alvo: arquivo final entre 200KB e 400KB
   - Nao comprimir se ja estiver abaixo de 200KB
   - Mostrar indicador de progresso durante compressao e upload
   - O comportamento JPEG existente nao deve ser alterado: parametro `outputType` deve ter `'image/jpeg'` como valor padrao

4. Upload para Supabase Storage no bucket `chamadas` (criado em Story 23.1):
   - Path: `{empresa_id}/{viagem_id}/{uuid}.webp`
   - `uuid` gerado no cliente com `crypto.randomUUID()`
   - Upload via server action `uploadChamada(formData: FormData)` em `app/(dashboard)/viagens/chamada-actions.ts`
   - Server action valida: tamanho maximo 5MB, tipos permitidos `image/jpeg,image/png,image/webp`, ownership da viagem pertence a empresa do usuario autenticado

5. Ausencia da chamada NAO bloqueia o submit do formulario de lancamento:
   - Badge "Chamada pendente" em cor amarela visivel nos cards de viagem em `components/viagens/ViagemList.tsx` (tanto na visualizacao mobile quanto na tabela desktop)
   - Badge aparece quando: `viagem.status IN ('planejada', 'em_andamento')` E nenhum registro existe em `foto_chamada` para aquela `viagem_id`
   - Texto do badge: "Chamada pendente"
   - Badge some automaticamente quando a chamada for adicionada (revalidatePath ja trata isso)

6. Botao secundario "Ja tenho a foto salva no celular" abre seletor de galeria (mesmo input sem atributo `capture`), posicionado abaixo do botao principal.

7. Imutabilidade enforced em dois niveis:
   - RLS do banco: motorista sem policy UPDATE nem DELETE em `foto_chamada` (implementado em Story 23.1)
   - Server action `uploadChamada`: nao expoe nenhuma rota de UPDATE ou DELETE para motorista; tentativas retornam HTTP 403
   - Interface: nao exibir nenhuma opcao de "Trocar foto" ou "Remover" para o role `motorista`

8. Validacao server-side na action `uploadChamada` (em `app/(dashboard)/viagens/chamada-actions.ts`):
   - Arquivo ausente, `viagem_id` ausente ou usuario nao autenticado: retornar erro descritivo em portugues
   - Viagem nao pertence a empresa do usuario autenticado: retornar `'Viagem nao encontrada'`
   - Role `motorista` E `viagem.status NOT IN ('planejada', 'em_andamento')`: retornar `'Nao e possivel adicionar chamada a uma viagem que nao esta em andamento'`
   - Role `dono` ou `admin`: nao verificar restricao de status (podem adicionar chamada a qualquer viagem)
   - Apos insert bem-sucedido: `revalidatePath('/viagens')` e `revalidatePath('/viagens/{viagem_id}')`

9. Mensagens de erro em linguagem humana, sem termos tecnicos:
   - Erro de upload: `'Nao conseguimos enviar a foto. Verifique a conexao e tente novamente.'`
   - Arquivo muito grande: `'A foto esta muito grande. Tente tirar com menos zoom ou escolha outra.'`
   - Formato nao suportado: `'Use apenas fotos (JPG, PNG ou WebP).'`

10. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #120. Epic L, P1, L/M. Depende de Story 23.1 (tabela `foto_chamada` + bucket `chamadas` existentes).

**Componente alvo:** `components/viagens/ViagemForm.tsx` — adicionar secao de upload da chamada. Posicionamento: apos os campos obrigatorios da viagem, antes do botao de submit.

**Server action nova:** criar `app/(dashboard)/viagens/chamada-actions.ts` espelhando `app/(dashboard)/gastos/comprovante-actions.ts`. Funcoes necessarias nesta story:
- `uploadChamada(formData: FormData): Promise<ChamadaActionResult>`

**Tipo novo:** criar `types/foto-chamada.ts` com `ChamadaActionResult` e `FotoChamada` espelhando `types/foto-comprovante.ts`.

**compress-image.ts — extensao necessaria:** o arquivo atual (`lib/utils/compress-image.ts`) tem output JPEG hardcoded (`'image/jpeg'` fixo no `canvas.toBlob`). Para chamadas o alvo eh WebP. Implementar assim:
- Adicionar parametro `outputType?: 'image/jpeg' | 'image/webp'` na funcao `compressImage`, com default `'image/jpeg'` para nao quebrar comportamento existente
- Ajustar `MAX_DIMENSION` para 1600 e qualidade inicial 0.8 quando o `outputType` for `'image/webp'` (ou receber esses valores como parametros adicionais)
- Nao remover nem alterar o comportamento JPEG — outras stories dependem dele (ex: Story 2.2)
- Ler o arquivo completo antes de modificar para entender a estrutura atual da iteracao de qualidade

**Badge "Chamada pendente" — implementacao:** o campo `chamada_pendente` nao existe no schema. Derivar na query de listagem de viagens: viagem sem nenhum registro em `foto_chamada` E `status IN ('planejada', 'em_andamento')`. Adicionar join ou EXISTS subquery na query em `app/(dashboard)/viagens/multi-queries.ts` (verificar onde a query de listagem vive). O badge deve aparecer tanto nos cards mobile quanto na tabela desktop em `components/viagens/ViagemList.tsx`.

**Fluxo de estado no cliente:**
```
idle → capturing (botao clicado) → previewing (foto selecionada) → uploading (confirmou) → done / error
```
Usar `useState` local no componente. Nao precisa de Zustand ou contexto global.

**Role check na server action:** usar `getCurrentUsuario()` de `lib/auth/get-user-role.ts`. O campo `usuario.role` retorna `'dono'`, `'admin'` ou `'motorista'`. Aplicar verificacao de status apenas quando `role === 'motorista'`.

**Imutabilidade — dono/admin podem sempre:** o dono/admin pode adicionar chamada a qualquer viagem, inclusive `concluida` ou `cancelada`. Apenas o motorista tem a restricao de status. A server action deve checar o role antes de aplicar a regra.

**Paths de referencia:**
- Form de viagem: `components/viagens/ViagemForm.tsx`
- Upload action referencia: `app/(dashboard)/gastos/comprovante-actions.ts`
- Tipos referencia: `types/foto-comprovante.ts`
- Compressao (ler antes de modificar): `lib/utils/compress-image.ts`
- Listagem e cards: `components/viagens/ViagemList.tsx`
- Query de listagem: `app/(dashboard)/viagens/multi-queries.ts`
- Auth helper: `lib/auth/get-user-role.ts` (`getCurrentUsuario()`)
- Logger: `lib/observability/logger.ts` (`logError`)
