## Story
**As a** @qa do FrotaViva,
**I want** uma bateria de testes automatizados e manuais que valide as 6 tools, o route handler, a UI e — o mais critico — **que garanta que o LLM sempre usa uma ferramenta em vez de inventar** e que **nao vaza dados entre tenants**,
**so that** o Assistente FrotaViva entre em producao com confianca de que respeita RLS e Article IV.

## Acceptance Criteria

1. **Cobertura de testes unitarios** para `lib/copilot/utils/period.ts` cobre os 6 casos minimos da 9.2 AC-5 + edge cases (string vazia, nome de mes invalido, periodo do futuro).
2. **Cobertura de testes de integracao** para as 6 tools (se ainda nao estavam na 9.2/9.3/9.4): cada tool e testada com mock de Supabase validando shape do output, filtros aplicados corretamente e cap `MAX_TOOL_ROWS` respeitado.
3. **Teste anti-alucinacao (critique R1):** teste de integracao que invoca o route handler `/api/assistente/chat` com cada uma das 5 perguntas-farol e verifica via spy que `streamText` chamou pelo menos 1 tool. O teste **FALHA** ruidosamente se o LLM responder sem chamar tool alguma. Como o LLM e nao-deterministico, este teste deve rodar **3 vezes** e aceitar 2/3 de sucesso como baseline (documentar na story caso o threshold precise calibracao).
4. **Teste cross-tenant:** teste de integracao (ou E2E) que:
   - Cria ou usa 2 contas demo (tenant A e tenant B) com dados fake distintos
   - Loga como tenant A, chama cada uma das 6 tools, e valida que **nenhum dado do tenant B** aparece no output
   - Idem reverso (logado como B, nao ve A)
   - Este teste e **hard fail** — qualquer vazamento bloqueia producao
5. **Teste de rate limit:** teste de integracao que envia 11 requests em menos de 60s do mesmo user e verifica que a 11a recebe 429 com JSON `{ error }` em pt-BR.
6. **Teste de degradacao do logger:** teste unitario que verifica que `logger.error()` nao lanca excecao mesmo se Sentry retornar erro (mock do transport).
7. **E2E manual** (documentado em `docs/stories/frotaviva-copilot/e2e-manual-checklist.md`):
   - Lista das 5 perguntas-farol executadas via UI em producao (ou preview)
   - Screenshot de cada resposta
   - Verificacao visual: markdown renderiza, streaming funciona, R$ formatado, sem jargao
   - Verificacao Lighthouse: `/` e `/dashboard` >= 95 nas 4 categorias
   - Verificacao rate limit: enviar 11 perguntas rapidas e confirmar banner amigavel
8. `npm test` passa com cobertura das novas tools >= 80% (medido com `npm run test:coverage`).
9. `npm run lint && typecheck && test && build` passam.

## Dev Notes

### Teste anti-alucinacao (R1 da critique)
O spy pode ser um mock do provider Gemini que intercepta as chamadas de tool. Se nenhuma tool for chamada apos 3 iteracoes, o teste falha. Alternativa mais simples: no test env, usar um modelo mock que **sempre** chama tool — mas isso perde valor de teste real. Preferir o spy sobre o real model se o free tier permitir.

Se o teste real contra Gemini consumir muita cota, pular para CI em branch main only. Documentar no checklist.

### Contas demo
Conforme decidido no GATE-1, o stakeholder ja tem varias contas demo com dados fake. Identificar 2 delas com dados distintos (caminhoes e gastos diferentes) para o teste cross-tenant. Credenciais vao em env vars de teste (nunca no repo).

### Threshold 2/3
LLMs sao nao-deterministicos. 2/3 de sucesso e um baseline razoavel para teste de aceitacao de integracao. Se calibracao mostrar que 3/3 e viavel, subir o threshold.

### Rate limit em testes
Se os testes consumirem o rate limit real, usar um user de teste dedicado e resetar entre runs (se o provider permitir) ou apenas tolerar que o teste so rode 1x por minuto no CI.
