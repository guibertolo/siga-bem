# Story 21.4: Validar ownership de motorista_id e caminhao_id em updateViagem e createFechamento

## Status

## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que `updateViagem` e `createFechamento` validem que `motorista_id` e `caminhao_id` pertencem a mesma empresa do usuario autenticado antes de persistir,
**so that** seja impossivel um atacante associar recursos de outra empresa a uma viagem ou fechamento (cross-tenant data poisoning).

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Em `updateViagem` (linha ~367-380 de `viagens/actions.ts`), antes de executar o UPDATE: se `motorista_id` estiver presente no payload, executar `SELECT id FROM motorista WHERE id = {motorista_id} AND empresa_id = {empresaId}` — abortar com erro 403 se nao encontrado.
2. Em `updateViagem`, se `caminhao_id` estiver presente no payload, executar validacao analoga para `caminhao` (AC: 1).
3. Em `createFechamento` (linhas ~231-237 de `fechamentos/actions.ts`), antes do overlap check: validar que `motorista_id` recebido pertence a empresa do usuario autenticado.
4. A validacao e feita via SELECT autenticado (nao service_role), aproveitando RLS como segunda camada.
5. Em caso de falha, a action retorna erro descritivo sem revelar dados da empresa alheia (ex: "Motorista nao encontrado" — nao "Motorista pertence a outra empresa").
6. Testes unitarios para os cenarios de ownership invalido (motorista_id de empresa B passado para usuario de empresa A).
7. `npm run lint && npm run typecheck && npm test` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #104 (H3 cyber-chief, M1 cyber-chief). Prioridade P1.

**Problema raiz (cross-tenant data poisoning):** Um usuario autenticado na empresa A pode enviar no payload um `motorista_id` que pertence a empresa B. Mesmo que a viagem seja da empresa A, ela ficaria vinculada a um motorista de outra empresa. Isso polui dados cross-tenant sem violar a RLS da tabela `viagem` diretamente.

**Ponto de ataque em `updateViagem`:**
```typescript
// viagens/actions.ts ~367-380 (area de vulnerabilidade)
const payload = {
  motorista_id: formData.motorista_id, // vem do frontend sem validacao de ownership
  caminhao_id: formData.caminhao_id,   // idem
  ...
};
await supabase.from('viagem').update(payload).eq('id', viagemId);
```

**Correcao padrao:**
```typescript
// Antes do UPDATE, para cada FK recebida:
if (formData.motorista_id) {
  const { data: motorista } = await supabase
    .from('motorista')
    .select('id')
    .eq('id', formData.motorista_id)
    .eq('empresa_id', empresaId)
    .single();

  if (!motorista) {
    return { error: 'Motorista nao encontrado' };
  }
}
```

**Reusar helper `assertOwnership` da Story 21.3:** se a 21.3 for implementada primeiro, usar `assertOwnership(supabase, 'motorista', motorista_id, empresaId)` e `assertOwnership(supabase, 'caminhao', caminhao_id, empresaId)`.

**Dependencia:** Story 21.3 (helper assertOwnership). Se executadas em paralelo, implementar a validacao inline e refatorar para usar o helper apos a 21.3 estar pronta.

**Ponto de ataque em `createFechamento`:**
O overlap check busca viagens do motorista em periodo. Se `motorista_id` nao for validado, o check pode retornar falso negativo (overlap nao detectado porque a query filtra por empresa_id mas o motorista_id e de outra empresa).
