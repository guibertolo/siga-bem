## Story
**Como** dono de cegonha usando o FrotaViva,
**Quero** um formulario rapido de acerto avulso que mostre o que vou lancar em preview e peca apenas confirmacao,
**Para que** eu lance bonus, descontos ou pagamentos pontuais em segundos sem preencher 10 campos de viagem.

## Acceptance Criteria

1. Uma nova server action `criarAcertoAvulso(input)` existe em `app/(dashboard)/fechamentos/actions.ts` (ou no repositorio criado na story 10.3 se ja tiver sido feita).
2. A server action valida entrada com Zod: `motorista_id` (string uuid obrigatorio), `valor` (number positivo obrigatorio), `descricao` (string 3-200 chars obrigatorio), `empresa_id` (string uuid, derivado do contexto do usuario autenticado).
3. A server action verifica que o motorista pertence a empresa do usuario (RLS + validacao explicita).
4. A server action cria um `fechamento_item` novo com `tipo = 'avulso'`, `referencia_id = NULL`, `descricao` = entrada, `valor` = entrada, e vincula a um fechamento do motorista com status `aberto`. Se nao existir fechamento `aberto` para o motorista, a action cria um novo fechamento automaticamente.
5. Se o motorista ja tem fechamento `aberto` no mes corrente, o acerto avulso e adicionado a esse fechamento.
6. A UI do formulario de acerto avulso (`app/(dashboard)/fechamentos/avulso/page.tsx` ou rota equivalente) pede apenas 3 campos: motorista (combobox ou select), valor (input numerico), descricao (input texto).
7. Ao preencher os 3 campos, um preview inline mostra: "Voce vai lancar R$X,XX para [nome motorista] com descricao '[descricao]'. Confirmar?".
8. Dois botoes: "Confirmar lancamento" e "Cancelar". Cancelar limpa o form sem submeter.
9. Apos confirmar, a UI mostra mensagem de sucesso e redireciona para o fechamento do motorista onde o acerto foi adicionado.
10. Se a action falhar, a UI mostra mensagem de erro amigavel sem quebrar o form.
11. `npm run typecheck && npm run lint && npm run build` passam.
12. Testes de integracao cobrem: criar acerto avulso com motorista valido / com motorista de outra empresa (deve falhar) / com valor negativo (deve falhar, descontos sao P2) / com descricao vazia (deve falhar).

## Dev Notes

- Dependencia dura: stories 13.1 (migration) e 14.3 (testes de fechamentos) DEVEM estar completas.
- Regra do usuario: sem popups. O preview e inline no mesmo card do form, nao modal.
- Publico 55+: labels em PT-BR claro, sem jargao. "motorista", "valor", "descricao" sao aceitos. Evitar "destinatario" ou "beneficiario".
- O form reduzido e uma decisao ja tomada pelo usuario (item da story). Nao reabrir discussao sobre quantos passos tem o fluxo.
- Valor negativo (desconto) e P2 (story separada). Nesta story, valor deve ser positivo.
- Se o motorista nao tem fechamento aberto, a action cria um fechamento novo. Atencao: criar fechamento dispara RLS e possivelmente triggers. Testar bem.
- Cuidado com a semantica: acerto avulso vs ajuste. Acerto avulso e pagamento pontual. Ajuste e correcao retroativa. Diferentes por design. Nao confundir.

## Testing

- Unit tests: validacao Zod como funcao pura.
- Integration tests:
  - Criar acerto avulso para motorista valido - sucesso.
  - Criar acerto avulso para motorista de outra empresa - falha por RLS.
  - Valor = 0 ou negativo - falha por Zod.
  - Descricao vazia ou > 200 chars - falha por Zod.
  - Motorista sem fechamento aberto - cria fechamento novo + adiciona acerto.
  - Motorista com fechamento aberto - adiciona no fechamento existente.
- Manual testing:
  - Criar acerto real em dev local, validar que aparece no fechamento certo.
  - Tentar criar com valor 0 e ver mensagem amigavel.
  - Tentar criar em mobile (iPhone SE).
