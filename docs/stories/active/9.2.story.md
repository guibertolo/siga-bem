   - **execute:** query em `caminhao` com `.eq('ativo', true)`, `.in('empresa_id', empresaIds)`, `.limit(MAX_TOOL_ROWS)`, retorna `{ caminhoes: [{ id, placa, modelo, ano }] }`.

7. Teste de integracao em `__tests__/copilot/tools/listar-caminhoes.test.ts` idem.
8. Nenhuma tool faz `fetch` externo. Nenhuma tool aceita `empresa_id` ou `user_id` no input (sempre derivados do context).
9. Todas as tools sao `async` e envolvem queries em try/catch com `logger.error` + re-throw com contexto (CLAUDE.md Error Handling).

## Dev Notes

### RLS-awareness
Toda tool deve usar `createClient()` do `@/lib/supabase/server` (nunca o service_role). Isso garante que cookies do request levam a sessao do usuario autenticado e RLS filtra automaticamente. Ainda assim aplicamos `.in('empresa_id', empresaIds)` como **belt-and-suspenders** caso o usuario esteja em modo multi-empresa (ver `lib/queries/multi-empresa.ts`).

### Timezone
`period.ts` deve usar timezone `America/Sao_Paulo` para que "este mes" corresponda ao mes civil brasileiro. Nao usar UTC diretamente.

### Subtotais por categoria — decisao de NAO reaproveitar lib/queries/gastos.ts
Analisei `lib/queries/gastos.ts` → `getGastos` / `getGastosTotals`. Nao reaproveitei porque aquelas funcoes assumem contratos de UI (paginacao, subtotais com icones/cor da categoria, shape `GastoListItemWithFoto`) que nao casam com o que o LLM precisa consumir. A tool T1 escreve a propria query mais enxuta, respeitando os mesmos principios (RLS SSR, centavos, lookup de categoria). Isso evita acoplamento cruzado entre o modulo `copilot/` e o modulo `gastos/` — se o schema mudar um dia, ambos evoluem independentemente.

### Mock Supabase thenable
Primeira iteracao dos testes usou `.limit().mockResolvedValue(...)` — falhou porque o builder Supabase e **thenable**: qualquer ponto do encadeamento pode ser awaited para disparar a query. Solucao: mock de chain que expoe um `.then` proprio e resolve com o resultado. Helper `createChain(result)` nos dois test files.
