## Story
**Como** desenvolvedor do FrotaViva,
**Quero** que cada dominio (fechamentos, viagens, BI, dashboard) tenha uma unica camada de repositorio em `lib/repositories/` que abstraia single vs multi-empresa,
**Para que** a duplicacao de ~4.450 linhas entre `actions.ts` e `multi-queries.ts` seja eliminada e o comportamento entre modos single e multi pare de divergir silenciosamente.

## Acceptance Criteria

1. Existe a pasta `lib/repositories/` com um arquivo por dominio: `fechamentos.ts`, `viagens.ts`, `bi.ts`, `dashboard.ts`.
2. Cada repositorio expoe funcoes de leitura que aceitam um contexto de empresa (`empresa_id: string` ou `empresa_ids: string[]`) e internamente escolhem a query apropriada.
3. A camada de server actions (`app/(dashboard)/{dominio}/actions.ts`) passa a delegar para o repositorio. Nao faz mais queries Supabase diretas.
4. A camada `multi-queries.ts` (renomeada na story 10.2) e reduzida a wrappers finos que apenas chamam o repositorio com o contexto multi.
5. Funcoes duplicadas entre `actions.ts` e `multi-queries.ts` do mesmo dominio sao unificadas. A contagem de linhas gemeas e reduzida em pelo menos 60%.
6. Testes de integracao existentes continuam passando. Se houver testes de fechamentos (story 14.3), eles continuam verdes apos o refactor.
7. Testes de integracao de RLS (story 14.2) continuam passando. Zero vazamento cross-tenant e zero regressao de policy.
8. `npm run typecheck && npm run lint && npm run build` passam.
9. Dev local smoke test confirma que BI, fechamentos, viagens e dashboard renderizam identicos ao estado pre-refactor em single e multi-empresa.
10. Um arquivo `docs/architecture/repositories.md` documenta o padrao da camada de repositorio e regras de extensao.

## Dev Notes

- **Dependencia dura:** stories 10.1, 10.2, 14.2, 14.3 DEVEM estar completas antes de comecar esta. Refatorar logica financeira sem testes e inaceitavel.
- Sugestao de padrao: cada funcao do repositorio aceita `{ empresaIds: string[] }` como contexto. Modo single passa array de 1 elemento, modo multi passa N. A query interna usa `.in('empresa_id', empresaIds)`.
- Fechamentos e a area mais critica: qualquer bug gera pagamento errado. Testes de story 14.3 cobrem transicoes de status, overlap de periodo, divisao por percentual. Rodar em todas as etapas.
- BI (`bi/actions.ts` com 2069 linhas) e o maior arquivo a refatorar. Considerar sub-quebra por dominio funcional (filters, kpis, margem) mas isso pode ser feito em P2 (story separada no backlog). Nesta story, foco e so unificar single vs multi.
- Se algum calculo de KPI ou margem mudar apos refactor, STOP. Investigar divergencia.
- NAO adicionar features novas nesta story. Refactor puro.
