## Story
**As a** responsavel pela seguranca do FrotaViva,
**I want** que o acesso direto ao Postgres do Supabase (portas 5432 e 6543) seja restrito via Network Restrictions a IPs conhecidos,
**so that** mesmo que uma connection string seja exposta, um atacante sem IP na allowlist nao consiga conectar ao banco de dados.

## Epic
Epic 22 — Infra Hardening.

## Origem
Addendum 2026-04-14, item #9

## Contexto Tecnico

Estado atual verificado via leitura e grep do projeto:

- `.env.example` contem: `NEXT_PUBLIC_SUPABASE_URL` (REST/PostgREST API), `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`. Nenhuma variavel de acesso direto ao Postgres (`DATABASE_URL`, `DIRECT_URL`, `SUPABASE_DB_URL`).
- Grep em todos os arquivos `.ts` e `.js` do projeto: zero ocorrencias de `DATABASE_URL`, `DIRECT_URL`, `SUPABASE_DB_URL`, `postgres://` ou `postgresql://`. A aplicacao acessa o Supabase exclusivamente via PostgREST (REST API), nunca via conexao direta pg.
- `supabase/` contem apenas `migrations/`. Migrations sao aplicadas via `supabase db push` (Supabase CLI) pelo desenvolvedor/CI, usando conexao direta ao Postgres — isso precisa ser preservado na allowlist.
- Esta story NAO altera codigo da aplicacao. E inteiramente configuracao de dashboard + documentacao.

## Acceptance Criteria

1. No Supabase Dashboard do projeto FrotaViva, a funcionalidade Network Restrictions (Settings > Database > Network Restrictions) e ativada com uma allowlist configurada contendo:
   - IP(s) do desenvolvedor principal (Guilherme) ou range CIDR correspondente.
   - IPs dos Vercel Edge Functions, obtidos conforme procedimento documentado no AC 4.
   - Nenhum bloco `0.0.0.0/0` (acesso irrestrito) permanece na allowlist apos a configuracao.

2. Apos a configuracao do AC 1, uma tentativa de conexao TCP direta na porta 5432 ou 6543 do host Postgres do Supabase a partir de um IP fora da allowlist resulta em `Connection refused` ou timeout de conexao (nao em erro de autenticacao — o bloqueio ocorre antes do handshake).

3. Apos a configuracao do AC 1, o comando `supabase db push` executado a partir da maquina do desenvolvedor (cujo IP esta na allowlist) conclui sem erro de conexao e aplica migrations normalmente.

4. Um arquivo `docs/runbooks/supabase-network-restrictions.md` e criado documentando:
   - **Como obter IPs do Vercel:** acessar https://vercel.com/docs/security/deployment-ip-addresses, copiar os ranges CIDR listados para Edge Network. Alternativamente: `dig A cname.vercel-dns.com` retorna IPs individuais (menos confiavel — preferir documentacao oficial).
   - **Como adicionar novo IP a allowlist** (ex: outro desenvolvedor, runner de CI): Supabase Dashboard > Settings > Database > Network Restrictions > Add rule > inserir IP ou CIDR > Save.
   - **Impacto em CI/CD:** se o pipeline de CI/CD executar `supabase db push`, o IP do runner (ex: GitHub Actions runner) deve estar na allowlist. IPs do GitHub Actions: https://api.github.com/meta (campo `actions`). Adicionar o range correspondente ou usar Supabase Access Token com `--db-url` que bypassa Network Restrictions via API (verificar suporte na versao atual do CLI).
   - **Procedimento de emergencia:** se o desenvolvedor mudar de IP (ex: viagem, novo provedor), acessar o Supabase Dashboard via interface web (que nao usa conexao pg direta) e atualizar a allowlist antes de rodar migrations.

5. O arquivo `docs/runbooks/supabase-network-restrictions.md` criado no AC 4 e referenciado no `docs/runbooks/` (criar o diretorio se nao existir).

## Technical Notes

- Network Restrictions no Supabase opera na camada de rede (firewall), bloqueando pacotes TCP antes do handshake SSL/TLS do Postgres. Por isso o comportamento observavel e `Connection refused` ou timeout, nao erro de autenticacao `password authentication failed`.

- O Supabase expoe duas portas para conexao direta ao Postgres:
  - `5432`: conexao direta (pooler desativado)
  - `6543`: connection pooler (Supavisor/pgBouncer)
  Network Restrictions bloqueia ambas.

- A conexao via REST API (`NEXT_PUBLIC_SUPABASE_URL`) usa HTTPS na porta 443 e NAO e afetada por Network Restrictions — a aplicacao continua funcionando normalmente em producao (Vercel) apos a configuracao.

- Vercel Edge Functions e Serverless Functions nao tem IPs fixos publicamente garantidos para sempre. A documentacao oficial do Vercel lista os ranges usados atualmente. Monitorar mudancas periodicamente ou usar a alternativa de allowlisting por token de servico se o Supabase CLI suportar.

- Supabase Free Tier pode ter limitacoes na funcionalidade Network Restrictions — verificar se o plano atual suporta antes de iniciar.

- Complexidade: S / Risco M (configuracao incorreta pode bloquear `supabase db push` e migrations em CI — testar AC 3 antes de encerrar).
