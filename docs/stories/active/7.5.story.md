## Story
**As a** sistema do FrotaViva,
**I want** garantir que usuarios com multiplas empresas sempre tenham uma empresa ativa selecionada antes de acessar o dashboard, e que a troca de empresa valide o vinculo antes de atualizar,
**so that** nenhum usuario acesse dados sem contexto de empresa definido e nenhum dado de empresa nao vinculada vaze.

## Acceptance Criteria

1. O middleware (`lib/supabase/middleware.ts` ou `middleware.ts`) intercepta rotas do dashboard: se o usuario autenticado tem `empresa_id IS NULL` e possui mais de 1 empresa vinculada, redireciona para `/selecionar-empresa`.
2. Se o usuario tem `empresa_id IS NULL` e possui exatamente 1 empresa, o middleware define automaticamente essa empresa como ativa (chama `fn_switch_empresa`) e continua para o dashboard — sem mostrar tela de selecao.
3. A server action `switchEmpresa(empresaId)` valida a existencia de vinculo ativo em `usuario_empresa` ANTES de executar o UPDATE. Se nao houver vinculo: retornar erro, NAO atualizar `usuario.empresa_id`.
4. Nao e possivel trocar para uma empresa com `ativo = false` no vinculo `usuario_empresa` — a funcao `fn_switch_empresa` rejeita vinculos inativos.
5. Nao e possivel trocar para uma empresa com `empresa.ativa = false` — a funcao `fn_switch_empresa` verifica se a empresa esta ativa.
6. O middleware NAO intercepta rotas publicas (`/login`, `/signup`, `/aceitar-convite`, etc.) — apenas rotas sob `/(dashboard)`.
7. Se o usuario nao esta autenticado e tenta acessar rota do dashboard, o middleware redireciona para `/login` (comportamento existente, nao alterar).
8. Apos troca de empresa bem-sucedida, `revalidatePath('/')` e chamado para invalidar o cache do Next.js, forcando re-fetch de todos os dados.
9. Logs de auditoria: toda troca de empresa bem-sucedida e registrada (ao menos no `console.log` server-side com `[switchEmpresa] usuario_id, empresa_destino`).

## Dev Notes

### Onde fica o middleware
Verificar se o projeto usa `middleware.ts` na raiz ou `lib/supabase/middleware.ts`. Seguir o padrao existente.

### Verificacao de empresa_id NULL no middleware
O middleware ja le o usuario da sessao Supabase. Adicionar verificacao:
```typescript
const usuario = await getCurrentUsuario(supabase)
if (!usuario.empresa_id) {
  const empresas = await getUserEmpresas(supabase)
  if (empresas.length > 1) {
    return NextResponse.redirect(new URL('/selecionar-empresa', request.url))
  } else if (empresas.length === 1) {
    // auto-switch
    await supabase.rpc('fn_switch_empresa', { p_empresa_id: empresas[0].empresa_id })
    // continuar
  }
}
```

### fn_switch_empresa — verificacao de empresa.ativa
Adicionar ao SQL da funcao:
```sql
-- Verificar que a empresa destino esta ativa
IF NOT EXISTS (SELECT 1 FROM empresa WHERE id = p_empresa_id AND ativa = true) THEN
  RAISE EXCEPTION 'Empresa nao esta ativa';
END IF;
```

### Seguranca: fn_switch_empresa e SECURITY DEFINER
A funcao executa com privilegios elevados mas valida o vinculo do usuario autenticado (`auth.uid()`). Nao e possivel fazer bypass via RPC sem estar autenticado.

### Dependencias
- Story 7.1: `fn_switch_empresa`, tabela `usuario_empresa`
- Story 7.2: server action `switchEmpresa`, query `getUserEmpresas`

### Testing
- Usuario autenticado sem empresa_id, 1 empresa: middleware faz auto-switch, dashboard carrega
- Usuario autenticado sem empresa_id, 2 empresas: redireciona para /selecionar-empresa
- Tentar `switchEmpresa` com empresaId de empresa sem vinculo: erro retornado
- Tentar `switchEmpresa` com empresa inativa (`ativo=false` no vinculo): erro
- Tentar `switchEmpresa` com empresa com `empresa.ativa=false`: erro
- Rota publica `/login`: middleware nao interfere
- Rota publica `/aceitar-convite`: middleware nao interfere
