## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que as policies do bucket `comprovantes` no Supabase Storage estejam codificadas em uma migration SQL versionada no git,
**so that** o drift entre o estado do dashboard (correto) e o repositorio (sem policies) seja eliminado e o estado de seguranca seja reproduzivel e auditavel.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Migration nova cria as policies corretas do bucket `comprovantes` replicando exatamente o que existe no Supabase Dashboard desde o FIRE Block.
2. Policy de SELECT: `bucket_id = 'comprovantes'::text AND (storage.foldername(name))[1] = (fn_get_empresa_id())::text` — somente o usuario da propria empresa acessa.
3. Policy de INSERT: mesma condicao de SELECT (AC: 2) — usuario so faz upload em pasta da sua empresa.
4. Policy de DELETE: mesma condicao de SELECT (AC: 2) — usuario so deleta na pasta da sua empresa.
5. Nao existe policy de UPDATE (alinhado com principio de imutabilidade de comprovantes).
6. A migration usa `CREATE POLICY ... ON storage.objects` com `WITH CHECK` em INSERT e `USING` em SELECT/DELETE.
7. Teste RLS cross-tenant negativo no Storage: usuario empresa_A nao consegue SELECT, INSERT ou DELETE em path de empresa_B.
8. `supabase db push` aplica a migration sem erros em ambiente local.
9. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #101 (C1 cyber-chief). Prioridade P0.

**Historico:** Commit `bb843da` executou DROP das policies antigas. Recriacao foi feita manualmente no Supabase Dashboard. Policies corretas confirmadas no dashboard. O drift git vs dashboard esta ativo desde entao — o repositorio nao reflete o estado seguro.

**Policies a replicar (conforme dashboard):**

```sql
-- SELECT (USING)
CREATE POLICY "comprovantes_select_empresa"
ON storage.objects FOR SELECT
TO authenticated
USING (
  bucket_id = 'comprovantes'::text
  AND (storage.foldername(name))[1] = (fn_get_empresa_id())::text
);

-- INSERT (WITH CHECK)
CREATE POLICY "comprovantes_insert_empresa"
ON storage.objects FOR INSERT
TO authenticated
WITH CHECK (
  bucket_id = 'comprovantes'::text
  AND (storage.foldername(name))[1] = (fn_get_empresa_id())::text
);

-- DELETE (USING)
CREATE POLICY "comprovantes_delete_empresa"
ON storage.objects FOR DELETE
TO authenticated
USING (
  bucket_id = 'comprovantes'::text
  AND (storage.foldername(name))[1] = (fn_get_empresa_id())::text
);
```

**SEM policy UPDATE** — comprovantes sao imutaveis por design.

**Funcao `fn_get_empresa_id()`:** retorna o `empresa_id` do usuario autenticado via `auth.uid()`. Ja existe no banco, nao criar novamente.

**Estrutura de path no bucket:** `{empresa_id}/{gasto_id ou viagem_id}/{filename}`. O primeiro segmento do path (foldername[1]) e o `empresa_id`, garantindo isolamento por empresa.

**Importante:** a migration deve verificar se as policies ja existem antes de criar (usar `DROP POLICY IF EXISTS` antes de `CREATE POLICY`) para ser idempotente.
