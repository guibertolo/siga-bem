## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que o cadastro de nova empresa seja feito exclusivamente via RPC `fn_create_empresa_with_owner` SECURITY DEFINER,
**so that** a policy aberta `auth.uid() IS NOT NULL` na tabela `empresa` seja removida e o onboarding seja atomico e auditavel.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Migration remove a policy RLS `auth.uid() IS NOT NULL` (ou equivalente) da tabela `empresa` para INSERT.
2. RPC `fn_create_empresa_with_owner(p_nome text, p_cnpj text, p_plano text, ...)` criada com `SECURITY DEFINER`.
3. A RPC executa atomicamente em uma transacao: INSERT em `empresa` + INSERT em `usuario_empresa` (vinculando o usuario autenticado como dono).
4. A RPC valida que `auth.uid()` nao e nulo antes de inserir (protecao basica).
5. A RPC retorna o `empresa_id` criado em caso de sucesso, ou lanca excecao em caso de erro.
6. O fluxo de onboarding (pagina/server action de cadastro de empresa) e atualizado para chamar a RPC em vez de INSERT direto.
7. Teste de integracao: onboarding completo via RPC cria empresa e vinculo dono na mesma transacao.
8. Teste negativo: INSERT direto em `empresa` por usuario autenticado sem a RPC e rejeitado (policy removida).
9. `supabase db push` aplica as migrations sem erros.
10. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #106 (H5 data-engineer). Prioridade P1.

**Problema raiz:** Policy `auth.uid() IS NOT NULL` em `empresa` para INSERT permite que QUALQUER usuario autenticado insira empresas diretamente na tabela, burlando o fluxo de onboarding. Alem disso, se o INSERT em `empresa` for bem-sucedido mas o INSERT em `usuario_empresa` falhar, o sistema fica em estado inconsistente (empresa sem dono).

**Solucao — RPC transacional SECURITY DEFINER:**

```sql
CREATE OR REPLACE FUNCTION fn_create_empresa_with_owner(
  p_nome          text,
  p_cnpj          text,
  p_plano         text DEFAULT 'essencial',
  p_nome_dono     text DEFAULT NULL
) RETURNS uuid
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_empresa_id  uuid;
  v_usuario_id  uuid;
BEGIN
  -- Lookup do usuario autenticado na tabela usuario (nao auth.uid direto)
  SELECT id INTO v_usuario_id FROM usuario WHERE auth_id = auth.uid();

  -- Validacao basica
  IF v_usuario_id IS NULL THEN
    RAISE EXCEPTION 'Nao autenticado';
  END IF;

  -- Criar empresa
  INSERT INTO empresa (nome, cnpj, plano)
  VALUES (p_nome, p_cnpj, p_plano)
  RETURNING id INTO v_empresa_id;

  -- Vincular dono
  INSERT INTO usuario_empresa (usuario_id, empresa_id, role)
  VALUES (v_usuario_id, v_empresa_id, 'dono');

  RETURN v_empresa_id;
END;
$$;
```

**SECURITY DEFINER:** a funcao roda com privilegios do owner (postgres), o que permite contornar a RLS em `empresa` (sem a policy aberta) e `usuario_empresa`. O `SET search_path = public` e obrigatorio para prevenir search_path injection.

**Remover policy aberta:** apos criar a RPC e atualizar o codigo de onboarding, dropar a policy `auth.uid() IS NOT NULL` de INSERT em `empresa`.

**Atualizacao no codigo TypeScript:** localizar onde o onboarding chama INSERT direto em `empresa`. Antes de implementar, rodar `grep -r "supabase.from('empresa').insert" app/` para identificar o call-site exato no fluxo de onboarding. Substituir por `supabase.rpc('fn_create_empresa_with_owner', { p_nome, p_cnpj, p_plano })`.

**Dev Note — localizacao do call-site de onboarding:** Executar `grep -r "supabase.from('empresa').insert" app/` para localizar a server action de cadastro antes de implementar. O path exato nao esta mapeado — o grep e obrigatorio antes de comecar.

**Granulacao de permissao pos-story:** INSERT em `empresa` so sera possivel via `fn_create_empresa_with_owner`. Se precisar de outros modos no futuro, criar novas RPCs especificas.
