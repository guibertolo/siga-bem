## Story
**As a** administrador de seguranca do FrotaViva,
**I want** que as policies RLS da tabela `gasto` incluam clausula `WITH CHECK` explicita para INSERT e UPDATE,
**so that** nenhum usuario possa inserir ou alterar gastos com `empresa_id` ou `motorista_id` de outra empresa, mesmo que a clausula USING esteja correta.

## Epic
Epic 21 — SEC (Seguranca e Integridade).

## Acceptance Criteria

1. Migration dropa as 2 policies FOR ALL atuais da tabela `gasto` e recria como 4 policies separadas por operacao (SELECT, INSERT, UPDATE, DELETE).
2. Policy de INSERT/UPDATE para role `motorista`: `WITH CHECK (empresa_id = fn_get_empresa_id() AND motorista_id = fn_get_motorista_id())`.
3. Policy de INSERT/UPDATE para role `dono` e `admin`: `WITH CHECK (empresa_id = fn_get_empresa_id())`.
4. Policies SELECT e DELETE recriadas com clausula USING correta — sem regressao de comportamento existente.
5. Teste de regressao obrigatorio: dono consegue SELECT e DELETE de gasto da propria empresa; motorista consegue SELECT de gasto proprio. Nenhum desses fluxos pode regredir apos a migration.
6. Teste RLS negativo com role motorista: usuario motorista nao consegue inserir `gasto` com `empresa_id` de outra empresa.
7. Teste RLS negativo com role motorista: usuario motorista nao consegue inserir `gasto` com `motorista_id` que nao e o seu proprio.
8. Teste RLS positivo com role dono: usuario dono consegue INSERT em `gasto` com `empresa_id` correto.
9. `supabase db push` aplica a migration sem erros em ambiente local.
10. `npm run lint && npm run typecheck` passam sem novos erros.

## Dev Notes

**Origem:** Addendum 2026-04-14, item #107 (M5 data-engineer). Prioridade P1.

**Problema raiz:** Policies RLS com apenas clausula `USING` (sem `WITH CHECK` explicito) em INSERT e UPDATE podem permitir que o usuario insira/atualize linhas com valores invalidos nos campos de isolamento (`empresa_id`, `motorista_id`). O Supabase aplica USING como WITH CHECK por padrao em INSERT, mas a pratica recomendada e ser explicito — especialmente quando o modelo de dados tem dois campos de isolamento dependentes.

**Diferenca USING vs WITH CHECK:**
- `USING`: filtra QUAIS linhas o usuario pode VER/DELETAR (SELECT, DELETE)
- `WITH CHECK`: valida os VALUES que o usuario pode INSERIR/ATUALIZAR (INSERT, UPDATE)

**Policies alvo (a recriar com WITH CHECK):**

Para role `motorista`:
```sql
-- INSERT
CREATE POLICY "gasto_insert_motorista"
ON gasto FOR INSERT
TO authenticated
WITH CHECK (
  empresa_id = fn_get_empresa_id()
  AND motorista_id = fn_get_motorista_id()
);

-- UPDATE
CREATE POLICY "gasto_update_motorista"
ON gasto FOR UPDATE
TO authenticated
USING (
  empresa_id = fn_get_empresa_id()
  AND motorista_id = fn_get_motorista_id()
)
WITH CHECK (
  empresa_id = fn_get_empresa_id()
  AND motorista_id = fn_get_motorista_id()
);
```

Para role `dono`/`admin`:
```sql
-- INSERT
CREATE POLICY "gasto_insert_dono"
ON gasto FOR INSERT
TO authenticated
WITH CHECK (
  empresa_id = fn_get_empresa_id()
  AND fn_get_user_role() IN ('dono', 'admin')
);

-- UPDATE
CREATE POLICY "gasto_update_dono"
ON gasto FOR UPDATE
TO authenticated
USING (empresa_id = fn_get_empresa_id() AND fn_get_user_role() IN ('dono', 'admin'))
WITH CHECK (empresa_id = fn_get_empresa_id() AND fn_get_user_role() IN ('dono', 'admin'));
```

**Funcoes necessarias:**
- `fn_get_empresa_id()` — ja existe
- `fn_get_motorista_id()` — verificar se existe; retorna `motorista_id` do usuario autenticado via `usuario_empresa` ou `auth.jwt()`
- `fn_get_user_role()` — verificar se existe (veja Story 21.5)

**Estrategia:** auditar as migrations existentes para entender o estado atual das policies de `gasto` antes de criar as novas. Usar DROP POLICY IF EXISTS para idempotencia.

**Impacto no codigo TypeScript:** nenhum. Erros de WITH CHECK serao retornados pelo Supabase como erro de permissao RLS e tratados pelo handler existente.
