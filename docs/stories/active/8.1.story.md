## Story
**As a** dono/admin do FrotaViva,
**I want** uma server action atomica que cria o auth user, o registro motorista, o registro usuario e o vinculo usuario_empresa em uma unica operacao,
**so that** ao ativar o toggle "Criar acesso", o motorista fique 100% operacional no sistema sem necessidade de passos manuais adicionais.

## Acceptance Criteria

1. Existe a server action `createMotoristaComConta(data: MotoristaComContaFormData)` em `app/(dashboard)/motoristas/actions.ts`.
2. A action valida os dados do motorista com o schema Zod existente e valida o campo `email` com `z.string().email()`.
3. A action gera uma senha temporaria criptograficamente segura de 12 caracteres contendo letras maiusculas, minusculas, numeros e pelo menos 1 caractere especial (`!@#$%`), usando `crypto.randomBytes` ou equivalente.
4. A action chama `adminClient.auth.admin.createUser({ email, password: senhaGerada, email_confirm: true })` para criar o auth user.
5. Se `createUser` falhar (email duplicado, rate limit, etc.), a action retorna erro imediatamente sem criar nenhum registro nas tabelas do projeto.
6. Apos criar o auth user com sucesso, a action insere um registro em `motorista` com todos os campos validados e `usuario_id` preenchido com o ID do `usuario` que sera criado no proximo passo.
7. Apos inserir o motorista, a action insere um registro em `usuario` com: `auth_id` = ID retornado pelo Supabase Auth, `empresa_id` = empresa_id do usuario autenticado, `role = 'motorista'`, `nome` = nome do motorista, `email` = email informado, `motorista_id` = ID do motorista criado no passo anterior.
8. Apos inserir o usuario, a action insere um registro em `usuario_empresa` com: `usuario_id`, `empresa_id` e `role = 'motorista'`.
9. Apos todas as insercoes, a action atualiza `motorista.usuario_id` com o ID do usuario criado e `usuario.motorista_id` com o ID do motorista criado (vinculo bidirecional).
10. Se qualquer etapa apos a criacao do auth user falhar, a action chama `adminClient.auth.admin.deleteUser(authUserId)` para rollback e retorna erro descritivo.
11. Em caso de sucesso, a action retorna `{ success: true, motorista, credenciais: { email: string, senha: string } }`.
12. A action so pode ser chamada por usuarios com role `dono` ou `admin` (usa `requireRole(['dono', 'admin'])` existente).
13. A action exporta tambem `createMotorista` sem alteracoes (backward compatibility preservada).
14. O tipo `MotoristaComContaFormData` e exportado de `types/motorista.ts` com todos os campos de `MotoristaFormData` mais `email: string` e `criar_conta: true`.

### Admin Client

```typescript
import { createClient } from '@supabase/supabase-js';
export const adminClient = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,

```
A variavel `SUPABASE_SERVICE_ROLE_KEY` deve existir no `.env.local` (nunca expor no client).

### Atomicidade via Try/Catch em Cascata
O Supabase JS SDK nao suporta transacoes cross-schema. A estrategia e:
```
try {
  1. createUser(auth) -> authUserId
  try {
    2. INSERT motorista (usuario_id = null provisoriamente)
    3. INSERT usuario (motorista_id = motoristaId)
    4. INSERT usuario_empresa
    5. UPDATE motorista SET usuario_id = usuarioId
    6. UPDATE usuario SET motorista_id = motoristaId
  } catch {
    adminClient.auth.admin.deleteUser(authUserId) // rollback
    throw
  }
} catch (authError) {
  return error // sem rollback necessario
}
```

### Geracao de Senha
Usar `crypto` nativo do Node.js:
```typescript
import crypto from 'crypto';
function gerarSenhaTemporaria(): string {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZabcdefghjkmnpqrstuvwxyz23456789!@#$%';
  const bytes = crypto.randomBytes(12);
  return Array.from(bytes).map(b => chars[b % chars.length]).join('');
}
```
Evitar caracteres confusos (0, O, l, 1, I) para facilitar leitura pelo motorista.

### Sequencia de IDs
O `motorista.usuario_id` so fica disponivel apos o INSERT em `usuario`. Inserir motorista primeiro com `usuario_id = null` e atualizar depois via UPDATE e aceitavel. Alternativa: gerar UUID no JS antes do INSERT.

### Schema Existente
A action `createMotorista` atual em `app/(dashboard)/motoristas/actions.ts` ja tem a validacao Zod e o `requireRole`. Reusar o schema `motoristaSchema` existente para nao duplicar logica.

### Tabela usuario_empresa
Criada na Story 7.1. Verificar se a migration ja foi aplicada antes de implementar.

### IMPORTANTE: Senha retornada em texto plano
A senha temporaria e retornada para o frontend UMA VEZ para exibicao no modal. Ela NAO deve ser armazenada no banco (o Supabase Auth ja armazena o hash). Documentar isso como decisao explicita no codigo.
