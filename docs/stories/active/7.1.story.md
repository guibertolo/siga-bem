## Story
**As a** desenvolvedor do FrotaViva,
**I want** criar a tabela `usuario_empresa` com RLS, as funcoes SQL `fn_switch_empresa` e `fn_get_user_empresas`, fazer backfill dos vinculos existentes e atualizar `fn_get_user_role()` para ler da nova tabela,
**so that** o sistema suporte multiplas empresas por usuario com roles independentes por empresa, sem quebrar nenhum usuario ou policy existente.

## Acceptance Criteria

1. Tabela `usuario_empresa` criada com colunas: `id UUID PK`, `usuario_id UUID NOT NULL REFERENCES usuario(id) ON DELETE CASCADE`, `empresa_id UUID NOT NULL REFERENCES empresa(id) ON DELETE CASCADE`, `role usuario_role NOT NULL DEFAULT 'admin'`, `ativo BOOLEAN NOT NULL DEFAULT true`, `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
2. Constraint UNIQUE `uq_usuario_empresa (usuario_id, empresa_id)` adicionada.
3. Indices criados: `idx_ue_usuario ON usuario_empresa (usuario_id) WHERE ativo = true` e `idx_ue_empresa ON usuario_empresa (empresa_id) WHERE ativo = true`.
4. Backfill executado: `INSERT INTO usuario_empresa (usuario_id, empresa_id, role) SELECT id, empresa_id, role FROM usuario WHERE empresa_id IS NOT NULL ON CONFLICT DO NOTHING`. Todos os usuarios existentes ficam com vinculo ativo na sua empresa atual.
5. RLS habilitada na tabela `usuario_empresa` com 5 policies: `ue_select_own` (usuario ve seus proprios vinculos), `ue_select_empresa` (dono/admin da empresa ve vinculos da empresa), `ue_insert_dono` (dono pode vincular usuarios), `ue_update_dono` (dono pode alterar role/ativo), `ue_delete_dono` (dono pode remover vinculos, exceto o proprio).
6. `fn_get_user_role()` atualizada para ler o role de `usuario_empresa` com JOIN em `usuario`: `SELECT ue.role FROM usuario u JOIN usuario_empresa ue ON ue.usuario_id = u.id AND ue.empresa_id = u.empresa_id WHERE u.auth_id = auth.uid() LIMIT 1`. Funcao permanece com mesmo nome e assinatura.
7. Funcao `fn_switch_empresa(p_empresa_id UUID) RETURNS BOOLEAN` criada como SECURITY DEFINER: valida vinculo ativo em `usuario_empresa`, atualiza `usuario.empresa_id` e `usuario.role` para a empresa destino atomicamente.
8. Funcao `fn_get_user_empresas() RETURNS TABLE (empresa_id, razao_social, nome_fantasia, cnpj, role, is_active)` criada como SECURITY DEFINER: retorna todas as empresas ativas vinculadas ao usuario autenticado, com flag `is_active = (ue.empresa_id = u.empresa_id)`.
9. Coluna `usuario.empresa_id` alterada de `NOT NULL` para nullable via `ALTER TABLE usuario ALTER COLUMN empresa_id DROP NOT NULL`. Coluna `ultima_empresa_id UUID REFERENCES empresa(id)` adicionada a `usuario`.
10. `fn_get_empresa_id()` NAO alterada — continua retornando `usuario.empresa_id` sem modificacao. RLS de todas as 12 tabelas existentes funciona sem nenhuma alteracao.
11. Testes de backward compatibility: queries existentes em viagem, gasto, motorista, caminhao retornam dados corretos apos migration.
12. Script de rollback documentado no arquivo de migration.
13. Types TypeScript atualizados: interface `UsuarioEmpresa` criada em `types/usuario-empresa.ts`. Interface `Usuario` em `types/usuario.ts` atualizada com `empresa_id?: string | null` e `ultima_empresa_id?: string | null`.

## Dev Notes

### Decisao Arquitetural — empresa_id como empresa ativa
`usuario.empresa_id` muda de semântica: deixa de ser "empresa unica do usuario" e passa a ser "empresa ativa no momento". Esta decisao e a base de toda a feature multi-empresa. Ver `docs/architecture-multi-empresa.md` secao 9 para o racional completo.

### fn_get_empresa_id() NAO muda
A funcao `fn_get_empresa_id()` continua identica. Ela ja retorna `usuario.empresa_id` — que agora e a empresa ativa. As 46 referencias a esta funcao em 12 migrations de RLS ficam intocadas.

### Sincronizacao de role na troca
`fn_switch_empresa()` deve sincronizar `usuario.role` com `usuario_empresa.role` da empresa destino:
```sql
UPDATE usuario
SET empresa_id = p_empresa_id,
    role = (SELECT role FROM usuario_empresa
            WHERE usuario_id = v_usuario_id AND empresa_id = p_empresa_id)
WHERE id = v_usuario_id;
```
Isso garante que `fn_get_user_role()` funcione tanto pela funcao nova (JOIN) quanto pelo campo legado.

### DEFAULT de role em usuario_empresa
Usar `DEFAULT 'admin'` na coluna (igual ao definido no PRD/architecture). O backfill usa o `role` existente de `usuario`, entao nao depende do default.

### Coluna role legada em usuario
NAO remover `usuario.role` nesta story. O campo continua sendo sincronizado por `fn_switch_empresa`. Deprecacao planejada para sprint futura.

### Arquivo de migration
