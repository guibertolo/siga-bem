## Story
**As a** desenvolvedor do FrotaViva,
**I want** criar o route handler `/api/assistente/chat` que autentica o usuario, aplica rate limit, invoca `streamText` com as 6 tools e retorna streaming response, alem de instalar um logger (Sentry free tier com fallback gracioso para console),
**so that** a UI do Assistente (9.6) tenha um endpoint funcional e a feature possa ser exercitada end-to-end pela primeira vez.

## Epic
Epic 9 — Assistente FrotaViva. Depende de 9.1, 9.2, 9.3, 9.4.

## Acceptance Criteria

1. Arquivo `lib/copilot/logger.ts` criado exportando `logger = { info, warn, error }`. Cada metodo tenta enviar via Sentry (`@sentry/nextjs` ja instalado) **dentro de um try/catch**. Se o envio falhar (cota estourada, erro de rede), **degrada para `console`** sem lancar. Logs incluem contexto `{ feature: 'assistente', ...extra }`.
2. Arquivo `lib/copilot/system-prompt.ts` criado exportando constante `SYSTEM_PROMPT` em pt-BR com as seguintes instrucoes nao-negociaveis:
   - "Voce e o Assistente FrotaViva. Responde em portugues do Brasil, linguagem simples, sem jargao tecnico."
   - "Use SEMPRE as ferramentas disponiveis para consultar dados. NUNCA invente valores, nomes, placas, datas ou numeros."
   - "Se o usuario pedir algo que nenhuma ferramenta cobre, explique com clareza que aquele dado nao esta disponivel."
   - "Formate valores monetarios como R$ X.XXX,XX. Centavos vem das ferramentas, voce divide por 100."
   - "Use tabelas markdown quando houver mais de 3 linhas de dados tabulares."
   - "Nunca exponha termos como 'LLM', 'prompt', 'token', 'ferramenta', 'API', 'tool' para o usuario."
   - "Se o usuario pedir um resumo sem especificar periodo, pergunte qual periodo antes de chamar a ferramenta."
3. Arquivo `lib/copilot/tools/index.ts` atualizado exportando `buildToolset(context: ToolContext)` que retorna objeto com as 6 tools no formato esperado pelo `tool()` do AI SDK v6, cada uma injetando `context` no `execute`.
4. Arquivo `app/api/assistente/chat/route.ts` criado como `export async function POST(request: Request)`:
   - Autentica via `getCurrentUsuario()` de `@/lib/auth/get-user-role`; se null, retorna `401` com JSON `{ error: 'Nao autenticado' }`.
   - Carrega `getMultiEmpresaContext()` para obter `empresaIds`.
   - Aplica `checkRateLimit(usuario.id)`; se bloqueado, retorna `429` com JSON `{ error: 'Limite de perguntas atingido. Tente de novo em X minuto(s).', retryAfterSeconds }`.
   - Constroi o toolset via `buildToolset({ supabase, usuario, empresaIds })`.
   - Invoca `streamText({ model: google('gemini-2.0-flash'), system: SYSTEM_PROMPT, messages, tools, maxSteps: 5, maxTokens: 1024 })`.
   - Retorna `result.toDataStreamResponse()`.
   - Todo o handler em try/catch global que loga via `logger.error` e retorna `500` com JSON `{ error: 'Erro ao processar sua pergunta. Tente novamente.' }` em caso de falha nao tratada.
5. Runtime do route handler = `nodejs` (nao Edge). Declarar `export const runtime = 'nodejs'`.
6. Cap `maxTokens: 1024` na resposta para proteger cota do free tier.
7. `maxSteps: 5` para evitar loop infinito de tool calls.
8. Teste manual: rodar `npm run dev`, logar no app com tenant demo, fazer `curl POST /api/assistente/chat` (ou teste via UI mock) com cada uma das 5 perguntas-farol e verificar:
   - a) Resposta chega com dados reais do tenant
   - b) Logs mostram pelo menos 1 tool call por pergunta
   - c) Valores monetarios formatados como R$ no texto final
9. `npm run lint && typecheck && test && build` passam.

## Dev Notes

### Primeira integracao real
Esta e a primeira story onde ha chamada real ao Gemini e ao rate-limit provider. Se env vars nao estiverem populadas corretamente (da 9.1), esta story falha. Verificar `.env.local` antes de rodar.

### Sentry sem pagamento
O projeto usa Sentry free tier. Se a cota estourar, o `logger.ts` degrada para `console.log` sem quebrar a feature. Testar o fallback manualmente: desativar temporariamente `SENTRY_DSN` e confirmar que o logger ainda funciona via console.

### maxSteps = 5
O AI SDK permite o LLM chamar varias tools em sequencia (ex: primeiro `listar_caminhoes` para resolver um nome, depois `ranking_caminhoes_por_lucro`). 5 steps cobrem casos razoaveis sem risco de loop.

### Formato de erro
Todas as respostas de erro devem ser JSON com campo `error` em pt-BR amigavel (nunca stack trace cru). UI (9.6) vai ler esse campo e exibir.
