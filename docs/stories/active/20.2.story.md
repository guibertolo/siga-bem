# Story 20.2: Tipo de Viagem (carregada/vazia) + Permissao exclusiva do dono

## Story
**As a** dono de frota,
**I want** registrar viagens vazias (retorno/reposicionamento sem carga) como um tipo separado,
**so that** os calculos de rentabilidade, CPK e taxa de vazio reflitam a realidade operacional, diferenciando viagem produtiva de deslocamento sem receita.

## Context

Na operacao real de cegonheiros, e comum o motorista descarregar no destino e nao conseguir carga de retorno. O dono manda o caminhao voltar vazio pra buscar carga em outro lugar. Esse retorno consome combustivel, pedagio e desgaste sem gerar receita.

Hoje essas viagens ou nao sao registradas (km desaparece) ou sao registradas com `valor_total = 0` (indistinguivel de "esqueceu de preencher o frete").

### Cenario tipico

```
1. Dono combina frete SP → Betim (R$ 8.000)
2. Motorista entrega, descarrega
3. Nao tem carga de retorno em Betim
4. Dono decide: "volta vazio pra SP, tem carga la"
5. Motorista roda Betim → SP sem carga (800 km, ~R$ 1.500 em diesel + pedagio)
6. Custo real da operacao SP→Betim = frete R$ 8.000 - custos ida R$ 3.000 - custos volta vazia R$ 1.500 = lucro R$ 3.500
```

Sem registrar a volta vazia, o lucro aparece como R$ 5.000 (inflado em 43%).

### Permissao

Viagem vazia e decisao exclusiva do dono. O motorista nao cria viagem vazia sozinho porque:
- E uma decisao de custo/beneficio que so o dono toma
- Evita que motorista registre viagem vazia pra justificar km/combustivel indevido
- O gestor tambem pode criar (delega pelo dono), mas motorista nao

## Technical Notes

### Migration SQL

```sql
-- Criar enum
CREATE TYPE tipo_viagem_tipo AS ENUM ('carregada', 'vazia');

-- Adicionar coluna com default (nao-destrutiva)
ALTER TABLE viagem ADD COLUMN tipo tipo_viagem_tipo NOT NULL DEFAULT 'carregada';
```

### Validacao condicional no Zod

```typescript
// Schema muda baseado no tipo
const viagemSchema = z.object({
  tipo: z.enum(['carregada', 'vazia']).default('carregada'),
  valor_total: z.string().min(1),
  // ...
}).refine((data) => {
  if (data.tipo === 'carregada') {
    const centavos = parseBrlInputToCentavos(data.valor_total);
    return centavos !== null && centavos > 0;
  }
  return true; // vazia aceita 0
}, { message: 'Valor total deve ser maior que zero para viagem carregada', path: ['valor_total'] });
```

### Atribuicao de custo vazia a rota de ida

Logica em T13 (rentabilidade_rota):
1. Se viagem vazia tem `viagem_ida_id` preenchido, atribuir custos a rota da ida
2. Se nao tem vinculacao, atribuir custos a rota inversa (destino→origem da vazia vira custo da rota origem→destino)
3. Se nao achar match, contabilizar como "custo operacional sem rota"

## Out of Scope

- Viagem parcialmente carregada (ex: volta com meia carga) — futuro, seria um terceiro tipo
- Calculo automatico de probabilidade de carga de retorno por rota — muito futuro
- Sugestao de rotas com carga de retorno — integracao com marketplaces de frete
